# gstack 使用示例：以《近距离共享相册》需求文档为项目基准

本文档与 `docs_gstack_Cursor使用指南.md` 配套，演示如何以 **`docs_共享相册App需求文档.md`** 作为**唯一需求真源**，在 Cursor 中按 gstack 流水线组织 AI 协作。

---

## 1. 让对话绑定需求文档

在 Cursor 新对话中：

1. **@ 引用**：`@docs_共享相册App需求文档.md`（若已有设计稿，可额外 `@技术设计文档.md`）。
2. 建议在项目根目录的 **`AGENTS.md`** 或 **`.cursor/rules`** 中写明约定，例如：规划、评审、测试阶段优先按 gstack 技能流程执行；需求以 `docs_共享相册App需求文档.md` 为准。

详见 `docs_gstack_Cursor使用指南.md` 第四节。

---

## 2. 标准流水线：提示词模板

以下顺序对应指南中的 **5.1 标准迭代流水线**，可直接复制到 Cursor 对话中使用。

### 2.1 想清楚需求与切口 → `/office-hours`

**适用场景**：需求文档已较完整，但需要收紧 MVP、明确取舍与技术验证项。

**示例提示：**

```text
请按 gstack 的 office-hours 流程执行。
需求唯一来源：@docs_共享相册App需求文档.md。
输出：1）最小可行 MVP（对照文档 7.1）；2）若只做 MVP 需从第二阶段暂缓的功能；3）WiFi Direct / 热点双模式下的用户决策树；4）三条最大技术风险与验证实验（每个实验 1 天内可完成）。
```

### 2.2 编码前评审（可分开或合并）

#### 工程与架构 → `/plan-eng-review`

```text
请按 plan-eng-review 执行。
基于 @docs_共享相册App需求文档.md 第 3 章（Clean + MVVM、NanoHTTPD、WebSocket、Room、分片上传、断线重连）。
请给出：模块边界图、群主/客户端数据流、WebSocket 消息与 HTTP API 的一致性、8 台设备并发与群主存储瓶颈、测试矩阵（对齐文档第 6 章）。
```

#### 设计计划（Compose / Material 3）→ `/plan-design-review`

```text
请按 plan-design-review 执行。
对照文档第 4 章（网格、详情、创建引导、设备列表、状态色与间距）。
输出：各屏信息架构、关键交互状态（空/错/加载）、与 M3 组件映射表，并标出容易做成「AI 味界面」的点及改写建议。
```

#### 一站式规划 → `/autoplan`

```text
请用 autoplan 自动串联适用评审。
输入：@docs_共享相册App需求文档.md。
目标：得到一份「可开工」的合并版计划（CEO/设计/工程取舍已决策），并列出仍需我人工拍板的条目。
```

### 2.3 实现过程中

| 场景 | 命令或说法 | 与需求文档的结合 |
|------|------------|------------------|
| 现象不明、需根因 | `/investigate` 或「请按 gstack investigate」 | 写明复现：双机 WiFi Direct、断网、群主进程被杀等，对照 2.1.2、5.2 |
| 一批改动完成后 | `/review` | 说明对比基准分支；重点：权限、删除/回收站、WebSocket 与本地 DB 一致性 |
| 安全与隐私 | `/cso` | 对照 5.4：最小权限、日志脱敏、无服务端收集与实现是否一致 |

### 2.4 有可访问 URL 时（Web 原型 / 内网演示）

`/browse`、`/qa`、`/qa-only` 依赖 **URL**。纯 Android 无网页时，可先提供极简 Web 管理台或静态演示页，或改为 `/review` + 手工验收清单（对照第 6 章）。

**QA 示例：**

```text
请按 qa 技能，对 http://<局域网地址>/相册演示 做标准验收。
验收依据：@docs_共享相册App需求文档.md 第 2.1（上传/浏览/下载/删除权限）与第 4.2 主要页面；输出缺陷列表与优先级。
```

**仅报告、不改代码：**

```text
请按 qa-only，对上述 URL 出报告，不要直接改仓库代码。
```

### 2.5 发布与收尾

| 步骤 | 命令 | 示例话术 |
|------|------|----------|
| 提 PR、同步测试与说明 | `/ship` | 「当前分支对齐文档 7.1 MVP，请按 ship 流程执行」 |
| 合并与部署验证 | `/land-and-deploy` | 需事先用 `/setup-deploy` 配置（见使用指南 5.6） |
| 发布后观察 | `/canary` | 提供生产环境 URL |
| 文档与版本说明同步 | `/document-release` | 「同步 README/架构说明与需求文档 3.x、7.x 里程碑」 |
| 周度复盘 | `/retro` | 「对照里程碑 7.1 周任务回顾本周交付」 |

## 3. 最小「单日可跑通」路径

若只想快速体验 gstack 与本文档的结合，**最少三步**：

1. **`/office-hours`** + `@docs_共享相册App需求文档.md` → 明确 MVP 与风险。
2. **`/plan-eng-review`** → 锁定 NanoHTTPD + WebSocket + Room 的接口与失败模式。
3. **`/review`**（有代码后）→ 对照需求查遗漏（如回收站保留期、删除权限仅上传者与群主等）。

---

## 4. 环境提醒（与使用指南一致）

- **Windows**：若 `/browse` 等浏览器类技能失败，在 gstack 仓库执行 `bun install && bun run build` 后重新 `./setup --host cursor`；并确保 Node 可用（见 `docs_gstack_Cursor使用指南.md` 第三节、第八节）。
- **斜杠菜单未出现技能**：可在对话中直接写「请按 gstack 的 xxx 流程执行」，并 @ 本机 `~/.cursor/skills/gstack-*` 下对应 `SKILL.md`（见使用指南 3.3）。
