## 1. 项目概述

### 1.1 项目背景

随着智能手机摄影能力的不断提升，用户在聚会、旅行、活动等场景中拍摄的照片数量日益增多。传统方式下，照片分享主要依赖社交媒体或即时通讯工具，但这需要稳定的互联网连接，且存在隐私泄露风险。在无网络或网络受限的场景下，如山区、海岛、地下场所或大型室内活动现场，用户之间分享照片变得困难重重。此外，使用微信、QQ等工具分享原图往往会被压缩，画质受损，而通过蓝牙传输速度慢且操作繁琐。

基于以上痛点，本项目旨在开发一款基于Android平台的近距离共享相册应用。该应用允许一台设备创建共享相册作为群主，其他设备通过**WiFi Direct 直连**或连接群主开启的**便携热点**进入同一局域网后访问该相册。**完整产品愿景**包括：上传、查看、下载、删除照片，以及标记与评论等协作能力。上述能力的**分期范围以 §1.4 与 §7 为准**；**MVP 不包含标记、评论、完整实时推送**，避免与 §5.2/§7.1 冲突。整个过程无需互联网连接，在设备间本地链路内完成传输，实现高速、便捷、安全的照片共享体验。

### 1.2 项目目标

本项目的核心目标是打造一款完全离线运行的多人照片共享工具，具体目标包括：实现零配置的照片共享体验，用户无需设置复杂的网络参数即可开始使用；支持最多8台设备同时连接到同一个共享相册，满足小型聚会或家庭场景的需求；提供流畅的照片浏览体验，支持高清原图和缩略图的高效加载；确保照片传输的高速性，充分利用WiFi Direct的理论传输速度；实现**在可接受延迟内**的照片库**最终一致**（MVP 以轮询+条件请求为主，**非**毫秒级实时；第二阶段引入更强实时能力）；**分期提供**标记与评论等协作能力（见 §1.4）；保障用户隐私，所有数据仅在参与设备间传输，不上传至任何服务器。

### 1.3 适用范围与约束

本应用面向Android 8.0（API级别26）及以上版本设备。选择Android 8.0作为最低版本主要考虑以下因素：WiFi Direct功能在该版本得到了更好的优化和稳定性提升；前台服务通知在该版本有更友好的展示方式；存储访问框架在该版本更加完善，用户授权流程更加清晰。应用设计假设用户设备具备WiFi和蓝牙功能，且设备之间距离在WiFi Direct的有效范围内（室内约30-50米）。

## 2. 功能需求规格

### 2.1 核心功能模块

#### 2.1.1 相册创建功能

相册创建是应用的核心入口功能，支持两种创建模式以适应不同的设备兼容性和使用场景。

**WiFi Direct创建模式**利用Android原生的WiFi P2P技术实现设备间的直接连接，无需任何中间网络基础设施。用户点击"创建相册"按钮后，应用会在后台初始化WiFi P2P管理器，并尝试创建一个P2P群组。作为群主的设备会广播其存在，允许其他设备发现并请求加入。这种模式的优点是连接建立后设备仍可访问原有的WiFi网络（在支持的情况下），适合需要同时保持网络连接的场景。

**便携热点创建模式**将创建者的设备作为WiFi热点运行，其他设备连接到这个热点后通过局域网访问相册内容。这种模式的兼容性更好，几乎所有Android设备都能连接热点，但创建者设备将无法连接其他WiFi网络。选择权交给用户，应用会展示两种模式的特点，让用户根据实际情况选择。

无论采用哪种模式，创建者需要设置相册名称（用于在设备发现列表中显示）和自己的昵称（用于标识上传者身份）。创建成功后，应用会显示群组信息，包括连接地址、服务端口等，供其他设备加入时使用。同时会生成一个简单的加入指引，帮助不熟悉技术的用户也能顺利加入。

#### 2.1.2 设备连接与发现功能

设备发现与连接是实现多人协作的基础功能，应用需要提供直观、高效的设备发现和连接流程。

**设备发现**采用主动扫描和被动等待两种机制。用户可以主动点击"发现设备"按钮启动扫描，此时应用会列出周围可发现的所有共享相册群组，每个条目显示群组名称、创建者昵称、信号强度和当前连接人数。扫描过程持续一定时间（如15秒），用户可以随时选择某个群组发起连接。

**被动等待**模式下，用户点击"等待连接"后，应用会在后台持续广播自己的存在信息，等待其他设备主动连接。这种模式适合创建者已经就位、参与者陆续加入的场景，如婚礼现场的宾客连接。

**成员加入群主服务（MVP）**须同时提供两种并列方式，并复用同一套「解析群主服务地址 → 建立会话 → HTTP 访问」逻辑，避免重复实现：

1. **发现列表自动带出**：成员通过 §2.1.2 所述「发现设备」扫描，在列表中选择目标共享相册后，由应用根据发现结果自动填充或推导出群主 NanoHTTPD 服务的访问信息（如 `http://host:port` 或等价形式），用户确认后加入。
2. **扫描二维码**：群主在创建成功后的群组信息/设备列表等区域展示**加入二维码**；成员使用应用内扫码（或系统扫码入口，依实现而定）解析二维码内容后加入。**载荷格式以附录 §8.5 为唯一规范**（与发现列表解析得到的 `baseUrl` 经同一套规范化后复用）。

**说明**：上文「扫描」指设备/相册发现或相机扫码两类不同入口；二者均为 MVP 范围，与「手动输入 IP:端口」无强制依赖（可作为后续兜底或调试能力另文约定）。

**可实现性约束（发现列表）**：「发现设备」所依赖的底层机制须在工程中**显式选型并文档化**（例如：在已建立的局域网内使用 **NSD（`NsdManager`）** 注册 `_sharedalbum._tcp` 服务并携带相册名/昵称/端口；Wi‑Fi Direct 仅负责链路而**不**保证自带「应用层相册名」发现）。验收时以：**同链路下列表能列出群主、选中后能拼出可访问的 `baseUrl`** 为准。

**连接管理**界面需要清晰展示当前已连接设备的列表，包括每台设备的昵称、连接时长、已上传照片数量等信息。用户可以主动断开某个连接，也可以查看各设备的详细信息。应用需要处理连接断开的情况：当某台设备意外离开时，**MVP** 通过 **HTTP 轮询成员列表/连接状态接口**、或 **定时重拉** 与 **请求失败重试** 更新 UI；**第二阶段**可改为 WebSocket 推送断线事件（见 §1.4）。

#### 2.1.3 照片浏览功能

照片浏览是用户最常用的功能，需要在性能和体验之间取得平衡。

**相册首页**采用网格布局展示所有照片，默认按上传时间倒序排列（最新的照片在前）。每行显示3张照片，照片以正方形缩略图展示，超出屏幕宽度时自动换行。**MVP** 不在缩略图上叠加标记角标（收藏/评分等），**第二阶段**再启用（见 §1.4）。当照片数量较多时，采用分页加载机制，每次加载20张照片，滑动到接近底部时自动加载更多。

**照片详情页**提供照片的全屏展示，支持捏合缩放和双击缩放操作。缩放后的图片可以通过手势平移查看细节。详情页底部展示照片信息，包括文件名、尺寸、上传者、上传时间、文件大小等。同时提供操作入口：**MVP** 含下载到本地、删除照片（需权限验证）；**收藏/标记/评论入口**归入 **第二阶段**（见 §1.4），MVP 可隐藏或置灰避免误承诺。照片切换支持左右滑动，滑动过程中预加载相邻照片，确保切换流畅。

**筛选与排序**：**MVP** 至少支持按上传时间升序/降序（与网格一致）；按上传者筛选、按标记类型筛选、按文件名排序等 **第二阶段** 提供（见 §1.4）。

#### 2.1.4 照片上传功能

照片上传是贡献内容的主要方式，需要兼顾易用性和可靠性。

**选择照片**通过系统照片选择器实现，用户可以从本地相册中选择一张或多张照片。选择界面支持实时预览，选中的照片会显示缩略图确认。选择完成后进入上传确认页面，显示选中的照片列表、预计上传时间（基于当前连接状态估算）。

**上传过程（MVP）**采用 **整文件 HTTP POST（multipart/form-data）**；上传进度显示总体进度条和当前文件进度；失败时整文件重试，任务进入队列并可由用户手动触发重试。**分片上传、断点续传**为 **第二阶段** 能力，须另立 API 约定（见 §1.4、§5.2）。

**后台传输（MVP 定义）**：允许使用 **前台服务（Foreground Service）+ 通知** 保障退到后台后的上传任务尽量完成；**不承诺**进程被系统终止后仍能恢复（第三阶段可加强）。

**上传完成**后，各成员在 **下一次列表轮询或刷新** 中看到新照片（**最终一致**，非毫秒级实时）。缩略图在群主设备生成，其他设备拉取缩略图用于列表；原图仅在查看详情或下载时获取。若上传失败（如连接中断），应用保存任务状态，连接恢复后提示继续上传。

#### 2.1.5 照片下载功能

照片下载允许用户将共享相册中的照片保存到本地设备。

**单张照片下载（MVP）**在照片详情页点击下载按钮即可触发。下载过程中显示进度指示，下载完成后通过系统通知提醒用户，同时自动打开照片应用查看（可选）。下载的照片保存到用户指定的相册目录（需要用户授权存储权限）。

**批量下载**、**下载目录模板（按上传者分文件夹）**、**已下载角标** 为 **第二阶段** 增强（见 §1.4）。**MVP** 可提供单一默认保存路径（系统相册/用户选定目录之一）。

**下载目录管理（MVP）**至少支持选择默认保存位置（含 MediaStore 合规实现）；高级规则第二阶段扩展。

#### 2.1.6 照片删除功能

照片删除需要谨慎处理，确保**权限清晰**与**后果告知**；**MVP** 无回收站，**不可逆**（见下）；**可逆性**由第二阶段回收站提供（见 §1.4）。

**删除操作**仅允许照片上传者和群主执行。点击删除后弹出确认对话框，明确提示删除后果。**MVP** 为 **永久删除**（无回收站）。确认删除后，群主端移除文件并将元数据标记为已删除（软删除标记可用于同步）。

**同步删除（MVP）**：通过 **HTTP API 删除** + 其他成员 **轮询列表/拉取增量** 合并删除状态；**第二阶段** 引入 WebSocket 推送删除事件（见 §1.4）。离线设备下次连接时通过列表同步获得删除结果。

**误删恢复（回收站）**为 **第二阶段** 功能（见 §1.4）；MVP 不实现回收站，以避免与里程碑不一致。

#### 2.1.7 照片标记功能（第二阶段）

下列功能**不属于 MVP**（见 §1.4、§7.1），本节描述**完整产品**能力，供第二阶段设计与实现引用。**以下子段落（收藏/评分/标签/评论）均为第二阶段行为描述，不作为 MVP 验收依据。**

照片标记为照片提供了丰富的元数据和社交属性。

**收藏标记**是最常用的标记类型，点击心形图标即可添加或移除收藏状态。收藏的照片会在相册中优先显示（可配置），支持筛选查看所有收藏照片。收藏状态实时同步，其他用户可以看到某人是否收藏了某张照片。

**评分标记**允许用户为照片打1-5星评分。评分界面采用简洁的星级选择器，点击对应星级即可完成评分。同一张照片允许多个用户评分，显示时计算平均分和个人评分。对于用户本人未评分的照片，显示"点击评分"提示。

**文字标签**允许用户为照片添加自定义文字标签，如标注照片中的人物、地点、事件等。标签支持自动补全，曾使用过的标签在输入时会作为建议显示。标签可以被所有用户查看和搜索。

**评论功能**允许用户在照片下发表文字评论，支持@提及其他用户。评论显示发表者的昵称和时间，点击用户名可以查看该用户上传的所有照片。评论实时同步，新评论到达时显示通知（可配置静音）。

### 2.2 系统功能模块

#### 2.2.1 用户管理功能

**昵称设置**是唯一必填的用户信息，用户首次使用应用时需要设置昵称，之后可在设置中修改。昵称用于标识照片上传者身份；**评论/@ 出现时的展示**为 **第二阶段**（见 §1.4）。

**头像设置**为可选项，用户可以从本地照片中选择一张作为头像，或使用系统默认头像。头像显示于设备列表等位置；**评论头像**为 **第二阶段**。

**在线状态（MVP）**：展示 **当前已连接成员**（以可实现为准，数据来源为 HTTP 轮询或群主维护的成员列表）；**「最近在线」精细化**可与 WebSocket 同步在 **第二阶段** 增强。

#### 2.2.2 连接管理功能

**连接状态指示**在应用各页面顶部持续显示当前连接状态。状态包括：未连接（灰色图标）、正在连接（黄色图标闪烁）、已连接（绿色图标）、连接异常（红色图标）。

**信号强度显示**对于WiFi Direct连接，显示各设备的信号质量指示。信号强度分为四个等级，用图标或颜色区分，帮助用户判断设备间的物理距离是否合适。

**手动重连**功能允许用户在连接不稳定时手动触发重连流程。重连前会显示当前连接信息，确认后清除现有连接并重新发现和加入群组。

**退出群组**功能用于主动离开当前群组。退出前须通过确认对话框**明确告知**：退出后本机将**清除与该群组相关的相册元数据（如 Room 中的投影）及应用内缓存的缩略图/临时文件**；**已通过本应用下载并保存到系统相册或用户指定目录的照片文件不予删除**。退出后无法再接收该群组的同步更新。

#### 2.2.3 设置与偏好功能

**通知设置**：**MVP** 至少包含 **上传任务完成/失败**、**下载完成**（若实现通知）、**作为群主运行中**（前台服务，见 §5.1）；**新评论/@ 提及** 等通知为 **第二阶段**（见 §1.4）。通知可配置静默模式。

**下载设置**包括默认下载路径选择、下载完成后的操作（仅保存、打开照片、打开相册）、是否自动下载原图（流量敏感场景建议关闭）。

**显示设置**包括缩略图大小选择（紧凑/标准/大图）、是否显示头像、默认排序方式、深色模式开关等。

**存储管理**显示应用已使用的存储空间，包括缓存、下载的照片等。用户可以一键清理缓存，或设置自动清理规则（如超过一定大小后自动清理最早的缓存）。

## 3. 技术架构设计

### 3.1 系统架构概述

本应用采用Clean Architecture分层架构与MVVM设计模式相结合的方案，确保代码结构清晰、职责明确、易于维护和测试。整体架构分为四个主要层次：表现层（Presentation Layer）、领域层（Domain Layer）、数据层（Data Layer）和连接层（Connection Layer）。

表现层负责用户界面的构建和交互逻辑的处理，使用Jetpack Compose实现声明式UI，通过ViewModel管理UI状态，利用Kotlin Flow与数据层进行响应式通信。领域层封装核心业务逻辑，**MVP** 以照片上传、删除、列表同步等用例为主；标记/评论用例 **第二阶段** 启用（见 §1.4）。数据层实现领域层定义的接口，包括本地数据库（Room）的访问、**HTTP 客户端（OkHttp 等）** 对群主服务的调用；**WebSocket** 为 **第二阶段** 同步通道，**不得**写进 MVP 必需依赖。连接层专注于 Wi‑Fi Direct、便携热点、局域网发现（如 NSD）等与「拿到群主 baseUrl」相关的实现。

各层之间通过接口进行通信，依赖方向从外向内，即外层可以依赖内层，内层不依赖外层。这种设计使得各层可以独立开发和测试，更换实现技术时不会影响其他层。例如，后续如果需要支持其他平台，只需在数据层实现对应的接口，领域层和表现层无需修改。

### 3.2 技术选型详细说明

#### 3.2.1 WiFi Direct API

WiFi Direct是本应用实现设备直连的核心技术，它允许Android设备在无需中间接入点的情况下直接建立WiFi连接。这项技术基于Wi-Fi Alliance制定的Wi-Fi Direct标准，在Android平台通过WifiP2pManager类提供编程接口。

从技术特性来看，WiFi Direct支持一对一和一对多连接拓扑，最多允许8台设备同时加入一个群组。设备在连接建立后会自动协商角色，其中一台设备成为群主（Group Owner），其他设备作为客户端加入。传输速度取决于设备硬件能力，理论最大值可达250Mbps，实际使用中通常在50-100Mbps区间，足以流畅传输高清照片。传输距离在无障碍环境下可达200米，室内环境受墙体遮挡影响，通常在30-50米内信号质量较好。

在应用中的具体使用流程如下：创建相册时，应用首先检测设备是否支持WiFi Direct，然后调用createGroup方法创建P2P群组。创建成功后，群主设备的WiFi芯片会广播服务发现帧，允许周围设备发现这个群组。加入相册时，加入设备调用discoverPeers方法扫描周围的可连接群组，扫描结果以列表形式展示给用户。用户选择要加入的群组后，应用调用connect方法发起连接请求，群主收到请求后进行响应，连接协商完成后建立TCP/IP层面的通信通道。

需要注意的是，WiFi Direct在不同Android设备和厂商定制系统上的兼容性存在差异。部分设备可能存在WiFi Direct功能异常或不稳定的情况，因此应用中实现了热点模式作为备选方案，当用户反馈WiFi Direct连接困难时，可以切换到兼容性更好的热点模式。

#### 3.2.2 NanoHTTPD嵌入式HTTP服务器

NanoHTTPD是一个极轻量级的嵌入式HTTP服务器库，整个库仅包含约1600行Java代码，jar包大小不足50KB。它完全基于Java标准库实现，无需任何第三方依赖，非常适合在Android应用内嵌运行。

NanoHTTPD在应用中的核心作用是将群主设备转变为一个本地照片服务器。当群主创建相册后，NanoHTTPD在指定端口（默认8080）启动监听，为所有加入者提供标准的HTTP访问接口。这种设计具有极好的跨设备兼容性，因为任何支持HTTP协议的设备都可以轻松接入。

HTTP服务的API端点设计如下：GET /api/photos 返回照片列表JSON，包含照片ID、文件名、上传者、尺寸等信息；**响应应携带 ETag（或等价的全局版本号/If-None-Match、sinceVersion 等增量机制）**，以便客户端在轮询时用条件请求减少流量（如返回 304 Not Modified）。POST /api/upload 接收照片文件上传，使用multipart/form-data格式；GET /api/photos/{id} 获取照片详细信息；DELETE /api/photos/{id} 删除指定照片；GET /photos/{filename} 下载照片原图文件；GET /thumbnails/{filename} 下载照片缩略图；POST /api/sync（或等价批量同步端点）**用于离线设备/第二阶段同步深化，MVP 可不实现**，以免扩大验收面（见 §1.4）。

服务器需要处理并发请求，当多个客户端同时上传或下载时，NanoHTTPD的多线程机制会为每个请求分配独立线程处理，确保服务响应不受影响。对于大文件传输，服务器支持Range请求头，允许客户端断点续传，这对不稳定的无线连接尤为重要。

#### 3.2.3 OkHttp WebSocket实时通信（第二阶段）

**说明**：**MVP 不以 WebSocket 为交付要件**（见 §1.4、§5.2）；本节描述 **第二阶段** 目标架构，供后续迭代。

OkHttp是Square公司开发的高效HTTP客户端库。OkHttp不仅支持传统HTTP请求，其内置的WebSocket实现可作为 **第二阶段** 实时同步通道。

在应用中的同步消息类型（第二阶段）可包括：PHOTO_ADDED、PHOTO_DELETED、MARKER_UPDATED、COMMENT_ADDED、USER_JOINED、USER_LEFT、PHOTO_LIST_SYNC 等。消息传输采用JSON格式。WebSocket 连接可建立心跳与自动重连策略。

#### 3.2.4 Room本地数据库

Room是Google官方推荐的Android数据库抽象层，它构建在SQLite之上，通过编译时注解处理器提供类型安全的数据库操作API。Room相比直接使用SQLite具有显著优势：编译时SQL验证可以避免运行时SQL错误；响应式查询支持通过LiveData或Flow返回数据；完善的迁移方案支持数据库结构升级；协程支持使得异步操作代码简洁易读。

**MVP** 至少包含 **photos** 表（或等价实体）用于照片元数据投影；**markers 表**在 **第二阶段** 启用（见 §1.4）。photos 表字段包括 id（UUID主键）、fileName、filePath、thumbnailPath、fileSize、mimeType、width、height、uploaderId、uploaderName、uploadTime、isDeleted（软删除标记）、lastModified。markers 表设计（第二阶段）可包括：id（自增主键）、photoId（外键）、markerType、value、userId、userName、createTime；与 photos 一对多，删除照片时级联删除标记。

数据库操作通过DAO（Data Access Object）接口暴露，接口方法使用Kotlin Flow返回响应式数据流。当数据库内容变化时（如收到同步消息更新），Flow会自动发射新数据，上层UI自动刷新。Repository层封装DAO操作，提供业务层面的数据访问接口给领域层使用。

#### 3.2.5 Glide图片加载库

Glide是Android平台公认最优秀的图片加载库之一，由Google官方推荐。它专注于流畅的列表滚动体验和低内存占用，支持网络图片、本地文件、资源Uri等多种图片来源，提供灵活的缓存机制和图片变换功能。

Glide在应用中的主要应用场景包括：相册首页的缩略图网格展示，Glide加载缩略图并自动管理内存缓存，确保快速滚动时不卡顿；照片详情页的高清原图加载，支持渐进式加载和过渡动画；照片上传时的缩略图预览；在下载管理器中显示下载中的图片缩略图。

Glide的缓存策略经过精心设计，包含两级缓存：内存缓存存储已加载的图片Bitmap对象，避免重复解码；磁盘缓存存储压缩后的图片文件，避免重复网络请求或文件读取。缓存大小可配置，默认使用动态计算值（基于可用内存的1/10）。

图片变换是Glide的另一项强大功能。应用使用Glide的transform方法实现圆角缩略图、自定义尺寸裁剪等效果。缩略图生成功能也依赖Glide的asBitmap方法加载并压缩图片。

#### 3.2.6 Jetpack Compose声明式UI

Jetpack Compose是Google发布的现代Android UI工具包，代表了Android界面开发的新方向。与传统XML布局不同，Compose采用完全声明式的编程范式，开发者只需描述UI应该呈现的状态，框架会自动处理UI的创建和更新。

Compose的主要优势体现在：代码简洁性，同样的界面效果使用Compose实现所需代码量显著少于XML加View系统的组合；状态驱动更新，UI组件自动响应状态变化，开发者无需手动处理视图刷新；实时预览，开发时可以直接在IDE中看到代码修改的UI效果；组件复用，Composable函数可以轻松提取为独立组件并在多处复用。

应用UI采用Material Design 3设计语言，所有组件使用Compose Material3库实现。主色调可选蓝色或绿色系，支持深色模式自动切换。图标使用Material Icons Outlined风格，保持视觉一致性。

主要页面包括：相册首页（GalleryScreen）展示照片网格；照片详情页（DetailScreen）展示单张照片和操作界面；相册创建页（CreateScreen）引导用户创建新相册；设备列表页（DeviceListScreen）显示已连接设备；设置页（SettingsScreen）配置应用偏好。

### 3.3 数据模型设计

#### 3.3.1 核心实体定义

下述 Kotlin 结构为**全量参考模型**；**MVP 实现至少包含 Photo**；`Marker`、`SyncEvent` 中与评论/标记相关部分 **第二阶段** 启用（见 §1.4）。

```kotlin
// Photo 照片实体
data class Photo(
    val id: String,                    // UUID，唯一标识
    val fileName: String,              // 原始文件名
    val filePath: String,              // 群主设备上的存储路径
    val thumbnailPath: String,         // 缩略图路径
    val fileSize: Long,                // 文件大小（字节）
    val mimeType: String,              // MIME类型，如 image/jpeg
    val width: Int,                    // 图片宽度（像素）
    val height: Int,                   // 图片高度（像素）
    val uploaderId: String,            // 上传者设备ID
    val uploaderName: String,          // 上传者昵称
    val uploadTime: Long,              // 上传时间戳（毫秒）
    val isDeleted: Boolean = false,    // 是否已删除（软删除）
    val lastModified: Long             // 最后修改时间戳
)

// Marker 标记实体
data class Marker(
    val id: Long = 0,                 // 数据库自增ID
    val photoId: String,               // 关联的照片ID
    val type: MarkerType,              // 标记类型
    val value: String?,                // 标记值（评分时为数字字符串，标签时为文字）
    val userId: String,                // 标记创建者ID
    val userName: String,              // 标记创建者昵称
    val createTime: Long               // 创建时间戳
)

// MarkerType 标记类型枚举
enum class MarkerType {
    FAVORITE,    // 收藏
    TAG,         // 文字标签
    RATING       // 评分
}

// User 用户实体
data class User(
    val id: String,                    // 设备唯一ID（首次安装时生成并持久化）
    val nickname: String,              // 用户昵称
    val avatarPath: String?,           // 头像路径（可选）
    val lastSeen: Long                 // 最后在线时间
}

// SyncEvent 同步事件
data class SyncEvent(
    val type: EventType,               // 事件类型
    val photoId: String? = null,        // 关联照片ID
    val data: String? = null,           // 事件数据（JSON字符串）
    val timestamp: Long,                // 事件时间戳
    val senderId: String                // 发送者ID
)

// EventType 事件类型枚举
enum class EventType {
    PHOTO_ADDED,
    PHOTO_DELETED,
    MARKER_ADDED,
    MARKER_REMOVED,
    COMMENT_ADDED,
    USER_JOINED,
    USER_LEFT,
    FULL_SYNC
}
```

#### 3.3.2 数据库表结构

photos表：id TEXT PRIMARY KEY（照片UUID）、fileName TEXT NOT NULL（文件名）、filePath TEXT NOT NULL（文件路径）、thumbnailPath TEXT（缩略图路径）、fileSize INTEGER（文件大小）、mimeType TEXT（MIME类型）、width INTEGER（宽度）、height INTEGER（高度）、uploaderId TEXT NOT NULL（上传者ID）、uploaderName TEXT（上传者昵称）、uploadTime INTEGER（上传时间）、isDeleted INTEGER DEFAULT 0（删除标记）、lastModified INTEGER（最后修改时间）。

markers表：id INTEGER PRIMARY KEY AUTOINCREMENT（自增ID）、photoId TEXT NOT NULL（照片ID，外键）、type TEXT NOT NULL（标记类型）、value TEXT（标记值）、userId TEXT NOT NULL（用户ID）、userName TEXT（用户昵称）、createTime INTEGER（创建时间）、FOREIGN KEY(photoId) REFERENCES photos(id) ON DELETE CASCADE。

users表：id TEXT PRIMARY KEY（设备ID）、nickname TEXT NOT NULL（昵称）、avatarPath TEXT（头像路径）、lastSeen INTEGER（最后在线时间）。

## 4. 界面设计规格

### 4.1 整体视觉风格

应用采用Material Design 3设计语言，这是一套Google发布的最新设计系统，强调个性化定制、包容性设计和动态色彩。界面设计追求简洁、直观、一致，让用户能够快速上手并高效完成任务。

**色彩体系**以蓝绿色系为主色调：**主色（Primary）偏蓝**、**辅色（Secondary 或 Tertiary）偏青绿**，辅以中性灰白色系；可启用 Material 3 动态取色，静态主题以上述蓝绿为种子色。深色模式下所有颜色自动适配。关键操作按钮使用主色调突出显示，次要操作使用文本按钮或图标按钮。**连接状态**宜采用**小面积图标/芯片**配合语义色（如 tertiary、error），避免大面积高饱和色块干扰阅读。状态语义：成功/在线、错误/删除、警告/待处理与 M3 语义色角色对齐。

**字体系统**使用Roboto字体族，包括Regular（正文）、Medium（标题）、Bold（强调）三种字重。标题使用Medium字重，字号24sp；正文使用Regular字重，字号16sp；辅助信息使用Regular字重，字号14sp，颜色使用次要文本色。

**图标系统**使用Material Icons Outlined风格，线条粗细一致，视觉重量均衡。图标尺寸标准为24dp（工具栏、操作按钮）、20dp（列表项图标）、16dp（内联图标）。图标颜色默认跟随文本颜色，支持着色器（tint）统一修改。

**间距系统**遵循8dp基准网格。组件内间距使用8dp或16dp，组件间间距使用8dp或16dp，页面边距统一使用16dp。卡片式组件使用12dp圆角，按钮使用8dp圆角，保持视觉一致性。

### 4.2 主要页面设计

#### 4.2.1 首页（相册浏览页）

首页是用户进入应用后的主界面，采用单Activity多Screen的导航架构。页面顶部是应用栏（TopAppBar），显示当前相册名称（如果已连接）或应用名称。应用栏右侧显示连接状态图标（绿色表示已连接）和用户头像（可点击进入设置）。

页面主体是照片网格（PhotoGrid），采用三列布局，每张照片显示为正方形缩略图。缩略图下方叠加显示标记数量（如收藏数和评分星）。照片之间保持4dp间距，网格与页面边缘保持16dp边距。缩略图加载时显示占位符（灰色背景），加载完成后淡入显示。

当照片数量较多时，网格支持惯性滚动。滚动到底部前会预加载下一页数据，避免等待感。快速滚动时图片加载可能被暂停，待滚动停止后恢复，这由Glide的生命周期集成自动处理。

页面右下角悬浮操作按钮（FAB），显示加号图标，点击进入照片选择器。如果未连接任何相册，FAB 或主导航应提供 **「创建相册」** 与 **「加入（发现/扫码）」** 入口（与 §2.1.2 MVP 一致）；**不得**仅保留创建而缺少加入路径。

#### 4.2.2 照片详情页

点击网格中的照片后进入详情页，照片全屏居中展示。顶部工具栏半透明悬浮，显示返回按钮和照片文件名。点击照片区域可以隐藏/显示工具栏，实现沉浸式浏览体验。

照片下方是信息面板，包含上传者头像和昵称、上传时间、文件尺寸和尺寸信息。面板背景使用表面色（Surface Variant），与照片区域形成对比。

面板下方是操作按钮栏：**MVP** 含下载、删除（权限与身份符合 §2.1.6）；**收藏/标记/评论** **第二阶段** 再提供（见 §1.4）。布局上可为第二阶段预留位置，但 MVP 验收以不展示或置灰为准。

底部展开区：**MVP** 不展示标记列表与评论区；**第二阶段** 展开。

#### 4.2.3 创建相册页

创建相册采用**单屏表单**：**相册名称为必填项**；**昵称**、**连接模式（WiFi Direct / 便携热点）** 置于 **「更多选项」可折叠区域**（默认折叠），展开后使用 Material OutlinedTextField 等组件填写或选择，减少误触与步骤割裂。若需弱化步骤感，可用简短辅助文案替代多步步骤指示器。

底部是操作按钮，创建按钮使用FilledButton样式，文字为"创建相册"。创建过程中按钮变为加载状态，显示环形进度指示器，禁止重复点击。

#### 4.2.4 设备列表页

设备列表页显示当前已连接的所有设备，以卡片列表形式展示。每张卡片包含设备头像（默认圆形图标）、昵称、连接时长、已上传照片数、最近活动时间。

卡片右侧可以显示连接质量指示器（三格信号图标）。点击卡片可以查看设备详情或执行操作（如移除连接、查看该设备上传的所有照片）。

页面顶部显示在线人数统计，如"3人在线"。**邀请加入（MVP）**：群主侧应提供**加入二维码**的展示与刷新（如适用），与「发现设备」并列作为成员接入路径之一（见 §2.1.2）。**第二阶段**可对邀请能力做增强（例如生成海报、站外分享、多端展示样式等），**不作为** MVP 必备条件。

### 4.3 交互反馈设计

**触摸反馈**所有可点击元素都有触摸涟漪（ripple）效果，点击时显示水波纹扩散动画。涟漪颜色使用当前颜色的叠加透明度，确保在任何背景上都清晰可见。

**操作确认**危险操作（如删除照片）需要二次确认，使用AlertDialog对话框。对话框标题使用Bold字重，说明文字使用Regular字重。操作按钮分为"取消"（文本按钮）和"确认"（FilledTonalButton或FilledButton，危险操作使用红色FilledButton）。

**加载状态**长时间操作显示进度指示器。确定性操作使用LinearProgressIndicator，显示已完成百分比。不确定性操作使用CircularProgressIndicator，表示正在处理但不显示具体进度。

**空状态**相册为空时显示友好的空状态插图和引导文字，如"还没有照片"配以"点击下方按钮上传第一张照片"的说明。插图使用品牌色调，文字使用次要文本色。

**错误状态**网络错误或操作失败时显示Snackbar提示。Snackbar出现在页面底部，自动消失（可选）或带有操作按钮。严重错误使用AlertDialog展示，提供"重试"或"取消"选项。

## 5. 性能与安全规格

### 5.1 性能指标要求

**应用启动时间**冷启动（从Launcher点击图标到首页可见）目标为2秒以内，热启动（从后台恢复）目标为0.5秒以内。启动过程中显示应用Logo或品牌页面（splash screen），但不应强制等待。

**照片加载性能**缩略图从本地缓存加载应在100ms内显示，从网络加载应在500ms内显示首帧。详情页原图：对 **JPEG** 可优先渐进式解码；对 **HEIC/PNG/WebP** 等格式以 **缩略图/占位图 → 全分辨率** 的分步加载实现等价体验（避免写死为仅 JPEG 渐进，导致不可实现）。

**列表滚动流畅度**照片网格滚动应保持60fps，测试设备为中端Android手机（如3年前的旗舰机）。滚动过程中允许适度的掉帧，但应避免明显卡顿。

**内存占用**应用在前台运行时内存占用目标为100-200MB。照片列表页面应能处理1000张照片而不出现内存问题，通过Glide的内存管理和图片回收机制实现。

**电量消耗**群主设备作为服务器运行时，电量消耗会高于普通使用。建议在群主设备上显示"作为群主运行中"的持续通知，提示用户保持充电状态。加入者设备应尽可能进入低功耗模式。

### 5.2 数据传输规格

**传输协议**照片上传使用HTTP POST协议，multipart/form-data格式。**第一阶段（MVP）以实现端到端可用为先：上传失败时支持整文件重试与任务队列；分片上传与断点续传在后续阶段深化实现并单独约定接口细节。** 同步消息在完整协作阶段使用WebSocket协议（JSON）；第一阶段可采用列表轮询配合上述 ETag/版本增量策略作为弱实时方案。

**文件大小限制**单张照片大小限制为50MB。超过限制时提示用户选择其他照片或压缩后重试。压缩选项允许用户在上传前选择是否压缩原图。

**传输加密**WiFi Direct连接本身支持WPA2加密，数据传输在链路层已加密。如需额外的端到端加密，可以考虑在应用层实现，但这会增加复杂度和性能开销。

**带宽考虑**高清照片原图可能达到数MB，上传时需要考虑带宽占用。**第二阶段** 可实现带宽自适应与「低带宽模式（仅缩略图）」等策略；**MVP** 以 **串行或有限并发（如 1～2）** 上传即可，并在 UI 上提示大文件可能较慢。

### 5.3 存储规格

**本地存储位置**照片文件存储在应用的内部存储目录（context.filesDir），与其他应用隔离，保证数据安全。数据库存储在同一目录的databases子目录下。缩略图存储在cache目录下，可以被系统清理。

**存储空间检查**上传照片前检查群主设备可用存储空间。阈值**不得写死为 5GB**（中端机常见总空间不足 5GB）：采用 **「可用空间低于 max(500MB, 单次上传上限×N)」** 或产品可配置阈值；低于阈值时提示清理；接近耗尽时阻止新上传。

**缓存管理**Glide磁盘缓存限制为100MB，超出后自动清理最旧的缓存条目。用户也可以在设置中手动清理缓存，或设置自动清理规则。

### 5.4 安全与隐私

**权限申请**应用需要申请以下权限：ACCESS_WIFI_STATE 和 CHANGE_WIFI_STATE 用于WiFi Direct操作；ACCESS_FINE_LOCATION 用于WiFi Direct设备发现（部分设备要求）；READ_EXTERNAL_STORAGE 或 READ_MEDIA_IMAGES 用于读取本地照片；WRITE_EXTERNAL_STORAGE 用于下载照片（Android 9及以下）；NEARBY_WIFI_DEVICES 用于Android 13+的附近设备访问。

权限申请遵循最小必要原则，运行时权限在需要时才申请。申请时提供清晰的权限用途说明，避免用户困惑。

**隐私保护**应用不收集任何用户数据到服务器，所有数据仅存储在参与设备本地。不使用任何分析SDK或追踪器。不在日志中打印敏感信息（如照片内容、用户输入）。

**数据隔离**每台设备的本地照片与群组照片严格隔离。本地照片不会自动上传，需要用户主动选择。离开群组后，群组照片从本地数据库清除，但用户已下载到本地相册的照片不受影响。

## 6. 测试计划规格

### 6.1 单元测试

**业务逻辑测试**覆盖领域层用例：**MVP** 优先覆盖照片上传、删除、列表同步；**标记/评论**用例在 **第二阶段** 补齐（见 §1.4）。使用 Mock 数据隔离外部依赖，确保测试的确定性。

**数据层测试**覆盖Repository实现和DAO方法。使用Room提供内存数据库进行测试，避免操作真实文件系统。测试数据增删改查的基本功能和边界条件。

**ViewModel测试**覆盖所有ViewModel的状态转换逻辑。使用Fake Repository或Mockito Mock对象模拟数据源。验证UI状态在各种操作后是否符合预期。

### 6.2 集成测试

**WiFi Direct集成测试**测试设备发现、连接建立、连接断开的完整流程。使用两台真实设备进行测试，或使用Roblectric框架模拟WifiP2pManager行为。

**HTTP服务器集成测试**测试照片上传、下载、列表查询等API。使用MockWebServer模拟客户端请求，验证服务器响应是否符合预期。

**端到端同步测试**：**MVP** 覆盖连接、上传、列表一致、删除、下载；**标记/评论**在 **第二阶段** 增补用例。

### 6.3 兼容性测试

**设备兼容性**在不同厂商、不同Android版本的设备上测试WiFi Direct功能。重点测试华为、小米、OPPO、VIVO、三星等主流厂商设备。记录各设备的兼容性问题，尝试提供workaround。

**网络环境测试**在多种WiFi Direct设备密度环境下测试，包括周围存在多个WiFi Direct群组、存在干扰源等情况。

**压力测试**模拟8台设备同时连接、同时上传照片的高负载场景，测试应用的稳定性和性能表现。

### 6.4 性能测试

**启动性能测试**使用Android Profiler和 systrace 工具测量启动时间，识别耗时操作并优化。

**内存分析**使用LeakCanary检测内存泄漏，使用Android Profiler分析内存分配热点。

**电量分析**使用Battery Historian分析应用运行期间的电量消耗，识别异常高耗电操作。

## 7. 项目里程碑规划

### 7.1 第一阶段：核心功能（MVP）

**目标**：实现基本的相册创建和照片共享功能，覆盖主流机型的**可达链路**（直连 + 热点兜底）。

**交付物**：可运行的应用原型；群主端支持 **WiFi Direct 与便携热点两种创建/组网方式**（与 §2.1.1 一致）；成员端在已建立链路下须同时支持 **通过发现列表加入群主服务** 与 **通过扫描群主二维码加入**（见 §2.1.2、§4.2.4）。在已连接群主服务后完成 **HTTP 列表（支持 ETag/版本增量，见 §3.2.2）**、**上传（第一阶段为整文件上传 + 失败重试，见 §5.2）**、**缩略图与原图浏览**、**单张下载**及 **§4.2** 所要求的基础 UI 骨架（含未连接态的「加入 / 发现设备 / 扫码」入口，见创建与首页相关规格）。

**功能范围**：双模式相册创建与加入、**设备发现与扫码加入（MVP 并列）**、连接管理（含权限与文案）、NanoHTTPD 照片服务、Room **photos 最小**元数据闭环、Compose 主流程（相册网格、详情、创建设备相关界面）、弱实时（轮询 + 条件请求）、**删除（永久，无回收站）**、**单张下载**；**不包含** WebSocket、标记/评论/回收站/批量下载、分片与断点续传（除 §1.4 已列外）；**邀请二维码的展示型增强**（海报、站外分享等）归入第二阶段（见 §7.2）。

**预计周期**：6-8周（含热点联调与双路径测试，可能接近区间上限）

**详细任务**：

第一周完成项目工程搭建，包括Android Studio项目创建、依赖库引入、代码规范制定。第二周完成**连接模块**：WiFi Direct（P2P 发现、建组、加入）与**便携热点创建/AP 配置**两条路径，保证至少一种路径在目标真机上稳定拿到群主 IP 与端口；**并明确发现列表与「加入二维码」的载荷格式与解析流程**，使成员端两条路径均接入同一 HTTP 客户端。第三周完成NanoHTTPD服务器，实现照片列表（含 **ETag 或版本号增量**）、上传与静态资源访问。第四周完成Room数据库设计和实现。第五周完成Jetpack Compose UI主界面（含 **创建页单屏 + 更多选项折叠**、**群主侧二维码展示与成员侧扫码入口**）。第六周完成照片浏览和下载功能。第七周进行单元测试和集成测试（**双模式**、**发现+扫码加入**与 HTTP 条件请求用例）。第八周进行Bug修复和性能优化。

### 7.2 第二阶段：协作功能

**目标**：实现完整的协作体验和实时同步

**交付物**：支持多人同时协作的完整应用

**功能范围**：实时同步、照片标记、评论功能、设备管理；**邀请加入的展示与分发增强**（如海报、站外分享、多端展示样式等，见 §4.2.4），**不作为** MVP 必备项。

**预计周期**：4-6周

### 7.3 第三阶段：体验优化

**目标**：提升应用稳定性和用户体验

**交付物**：生产就绪的应用版本

**功能范围**：离线支持、后台传输、错误恢复、UI细节打磨

**预计周期**：3-4周

### 8.4 MVP 验收一致性检查清单（研发自测）

在发布 MVP 前，应对照下列项逐项核对，**全部满足**方可视为与本文档一致：

1. **同步机制**：MVP 仅依赖 HTTP 列表轮询 + 条件请求；**未**将 WebSocket 列为必测项。
2. **上传**：整文件 POST + 重试队列；**未**将分片/断点续传列为 MVP 验收项。
3. **删除**：永久删除；**无**回收站；删除状态靠列表/增量同步合并。
4. **UI**：无标记角标、无评论入口（或置灰）；具备加入（发现 + 扫码）入口。
5. **存储**：群主阈值使用 §5.3 的可实施规则，**非**固定 5GB。
6. **发现**：已实现并文档化 NSD（或等价）方案，发现列表可还原 `baseUrl`。
7. **二维码**：群主生成的加入码、成员解析结果符合 **§8.5**（单行 HTTP URL、UTF-8、规范化后与发现列表一致）。

### 8.5 加入二维码载荷格式（MVP 规范）

本节为 **MVP 强制约定**，用于研发、联调与验收；与 `docs_共享相册App_MVP_implementation_plan.md` 中的技术方案应保持一致。

#### 8.5.1 载荷形态

| 项目 | 约定 |
|------|------|
| **编码** | **单行**文本，**UTF-8**（无 BOM）；**不得**使用多行、JSON 包裹（除非未来版本另立版本号并升级 §8.5）。 |
| **语义** | 载荷即群主 **HTTP 服务的根地址（baseUrl）**，成员端所有 API 请求均以此为前缀拼接（例如 `GET {baseUrl}api/photos` 的路径拼接规则由客户端统一处理，见 §3.2.2）。 |
| **内容** | 符合 [RFC 3986](https://www.rfc-editor.org/rfc/rfc3986) 的 **绝对 URI**，且 **scheme 为 `http`**（MVP 局域网明文服务；**第二阶段**若引入 TLS，可扩展为 `https` 并修订本节版本号）。 |

#### 8.5.2 URL 模板（固定）

**规范形式（推荐生成）：**

```text
http://<host>:<port>/
```

- **`<host>`**：IPv4 点分十进制 **或** 在该局域网下可解析的主机名（若使用主机名，须保证成员 DNS/mDNS 能解析，否则优先 IPv4）。
- **`<port>`**：与群主 NanoHTTPD 监听端口一致（文档默认 **8080**，若产品改为其他端口，二维码须同步带该端口）。
- **末尾 `/`**：可选；成员端解析后应 **规范化** 为「带单一 `/` 结尾的 baseUrl」或「无路径时等价根路径」，与发现列表拼出的字符串 **一致**，避免重复斜杠。

**显式要求（MVP）：** `host` 与 `port` **必须出现在载荷中**（**禁止**写成依赖默认端口 `80` 的省略形式），以降低弱网/定制系统下的歧义。

**示例（合法）：**

```text
http://192.168.49.1:8080/
http://192.168.43.1:8080
```

**示例（不合法 / MVP 须拒绝并提示）：**

```text
192.168.49.1:8080
https://192.168.49.1:8080/
http://192.168.49.1/
http://[::1]:8080/
```

说明：第 1 条缺少 scheme；第 2 条 scheme 非 MVP 约定；第 3 条缺少显式端口；第 4 条为 IPv6 字面量，MVP **不要求**支持（可选实现，未实现则提示错误）。

#### 8.5.3 解析与规范化（成员端）

1. 读取二维码原始字符串 → `trim()` 去首尾空白。  
2. 校验 **单行** 且非空。  
3. 使用标准库或经充分测试的 URL 解析器（如 OkHttp `HttpUrl`）解析；**scheme 必须为 `http`**，`host` 非空，`port` 为正整数且与约定范围一致（1–65535）。  
4. 将结果规范为内部类型 `baseUrl`（字符串或 `HttpUrl`），与 **NSD 发现** 路径生成的 `baseUrl` **共用同一入口**（见 §2.1.2）。  
5. 解析失败：展示可读错误（如「无效的加入二维码」），不静默降级。

#### 8.5.4 与发现列表的一致性

群主在 **NSD**（或文档约定的等价发现）中广播的端口、与二维码中的 `host:port`，在**同一时刻**应指向**同一 NanoHTTPD 实例**。验收建议：同一成员设备先 **发现加入** 再 **扫码加入**，所连 `baseUrl` 规范化后应一致（允许 host 形式不同但解析到同一服务时，以工程约定为准并在联调记录中备案）。

#### 8.5.5 预留（非 MVP）

- **Query**：如 `?v=1` 用于未来协议版本，**MVP 生成端可不输出**；成员端 **MVP 应忽略未知 query**（若存在），仅以 scheme/host/port/path 根为准。  
- **自定义 scheme**（如 `sharedalbum://`）：**不纳入 MVP**；若第二阶段引入，须新版本号并更新 §8.5。
