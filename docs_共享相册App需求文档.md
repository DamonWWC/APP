## 1. 项目概述

### 1.1 项目背景

随着智能手机摄影能力的不断提升，用户在聚会、旅行、活动等场景中拍摄的照片数量日益增多。传统方式下，照片分享主要依赖社交媒体或即时通讯工具，但这需要稳定的互联网连接，且存在隐私泄露风险。在无网络或网络受限的场景下，如山区、海岛、地下场所或大型室内活动现场，用户之间分享照片变得困难重重。此外，使用微信、QQ等工具分享原图往往会被压缩，画质受损，而通过蓝牙传输速度慢且操作繁琐。

基于以上痛点，本项目旨在开发一款基于Android平台的近距离共享相册应用。该应用允许一台设备创建共享相册作为群主，其他设备通过**WiFi Direct 直连**或连接群主开启的**便携热点**进入同一局域网后访问该相册。**完整产品愿景**包括：上传、查看、下载、删除照片，以及标记与评论等协作能力。

### 1.2 项目目标

本项目的核心目标是打造一款完全离线运行的多人照片共享工具，具体目标包括：
- 零配置的照片共享体验，用户无需设置复杂的网络参数即可开始使用
- 支持最多8台设备同时连接到同一个共享相册
- 提供流畅的照片浏览体验，支持高清原图和缩略图的高效加载
- 照片传输的高速性，充分利用WiFi Direct的理论传输速度
- **在可接受延迟内**的照片库**最终一致**（MVP 以轮询+条件请求为主，**非**毫秒级实时）
- **分期提供**标记与评论等协作能力（见 §1.4）
- 保障用户隐私，所有数据仅在参与设备间传输，不上传至任何服务器

### 1.3 适用范围与约束

本应用面向Android 8.0（API级别26）及以上版本设备。选择Android 8.0作为最低版本主要考虑以下因素：WiFi Direct功能在该版本得到了更好的优化和稳定性提升；前台服务通知在该版本有更友好的展示方式；存储访问框架在该版本更加完善，用户授权流程更加清晰。应用设计假设用户设备具备WiFi和蓝牙功能，且设备之间距离在WiFi Direct的有效范围内（室内约30-50米）。

## 2. MVP 定义（第一阶段，§7.1）

> **⚠️ MVP 边界声明**：以下 §2.1 是 MVP **唯一**功能范围，凡未列入的功能均为第二/三阶段。需求文档其他章节中出现的功能描述仅供参考，不作为 MVP 验收依据。

### 2.1.1 相册创建功能

**群主**需要设置相册名称（用于在设备发现列表中显示）和自己的昵称（用于标识上传者身份）。创建成功后显示群组信息，包括连接地址、服务端口等。

**创建模式（MVP）**：**WiFi Direct** 作为主路径，**便携热点** 作为备选/兼容性路径（见 §2.1.2）。

### 2.1.2 设备发现与连接功能

**成员加入群主服务（MVP）**须同时提供两种并列方式，并复用同一套「解析群主服务地址 → 建立会话 → HTTP 访问」逻辑：

1. **发现列表自动带出**：成员通过 **NSD（`NsdManager`）** 注册 `_sharedalbum._tcp` 服务发现群主，应用根据发现结果自动填充群主 NanoHTTPD 服务的访问信息（如 `http://host:port`），用户确认后加入。
2. **扫描二维码**：群主在创建成功后的群组信息区域展示**加入二维码**；成员使用应用内扫码解析二维码内容后加入。

**二维码载荷格式（强制约定，见 §8.5）**：
- 单行 UTF-8 文本，无 BOM
- 内容为群主 HTTP 服务的根地址（baseUrl），scheme 必须是 `http`，必须显式包含 host 和 port
- 规范形式：`http://<host>:<port>/`
- 禁止：缺少 scheme、缺少端口、https、IPv6 字面量

**可实现性约束**：Wi‑Fi Direct 仅负责链路层，**不**负责应用层相册名发现。NSD 服务 TXT 记录携带 `name=<相册名>`、`nick=<群主昵称>`、`ver=1`。

### 2.1.3 照片浏览功能

**相册首页**采用网格布局展示所有照片，默认按上传时间倒序排列。每行显示3张照片，照片以正方形缩略图展示。分页加载机制，每次加载20张照片。

**照片详情页**提供照片的全屏展示，支持捏合缩放和双击缩放操作。详情页底部展示照片信息：文件名、尺寸、上传者、上传时间、文件大小。

**筛选与排序（MVP）**：按上传时间升序/降序。**不提供**按上传者筛选、按标记类型筛选。

### 2.1.4 照片上传功能

**上传过程（MVP）**：
- 整文件 HTTP POST（multipart/form-data）
- 上传进度显示总体进度条和当前文件进度
- 失败时整文件重试，任务进入队列并可由用户手动触发重试
- **不提供**分片上传、断点续传（第二阶段）
- 单张照片大小限制为 **50MB**

**后台传输（MVP）**：使用 **前台服务（Foreground Service）+ 通知** 保障退到后台后的上传任务尽量完成；**不承诺**进程被系统终止后仍能恢复。

**同步方式（MVP）**：各成员在 **下一次列表轮询或刷新** 中看到新照片（**最终一致**，非毫秒级实时）。

### 2.1.5 照片下载功能

**单张照片下载（MVP）**：在照片详情页点击下载按钮即可触发。下载过程中显示进度指示，下载完成后通过系统通知提醒用户。下载的照片保存到用户指定的相册目录。

**不提供**批量下载、下载目录模板（按上传者分文件夹）、已下载角标（第二阶段）。

### 2.1.6 照片删除功能

**删除操作**仅允许照片上传者和群主执行。点击删除后弹出确认对话框，明确提示删除后果。

**MVP 为永久删除（无回收站）**。确认删除后，群主端移除文件并将元数据标记为已删除（软删除标记）。

**同步删除（MVP）**：通过 HTTP API 删除 + 其他成员轮询列表/拉取增量合并删除状态。离线设备下次连接时通过列表同步获得删除结果。

**不提供**回收站/误删恢复（第二阶段）。

### 2.1.7 连接状态与设备列表

**连接状态指示**：在应用各页面顶部持续显示当前连接状态。状态包括：未连接（灰色图标）、正在连接（黄色图标闪烁）、已连接（绿色图标）、连接异常（红色图标）。

**设备列表**：显示当前已连接的所有设备，以卡片列表形式展示。每张卡片包含设备昵称、连接时长、已上传照片数量等信息。

**不提供**信号强度显示（三格图标，第二阶段）、精细在线状态（WebSocket，第二阶段）。

### 2.1.8 用户管理功能（MVP 最小集）

**昵称设置**是唯一必填的用户信息，用户首次使用应用时需要设置昵称，之后可在设置中修改。昵称用于标识照片上传者身份。

**不提供**头像设置、评论头像（第二阶段）。

## 5. 技术架构设计

### 5.1 系统架构概述

本应用采用Clean Architecture分层架构与MVVM设计模式相结合的方案。整体架构分为四个主要层次：表现层（Presentation Layer）、领域层（Domain Layer）、数据层（Data Layer）和连接层（Connection Layer）。

表现层负责用户界面的构建和交互逻辑的处理，使用Jetpack Compose实现声明式UI，通过ViewModel管理UI状态，利用Kotlin Flow与数据层进行响应式通信。领域层封装核心业务逻辑。数据层实现领域层定义的接口，包括本地数据库（Room）的访问、HTTP 客户端（OkHttp）对群主服务的调用。连接层专注于 Wi‑Fi Direct、便携热点、局域网发现（NSD）等与「拿到群主 baseUrl」相关的实现。

### 5.2 技术选型

| 技术 | 用途 |
|------|------|
| **Kotlin + Jetpack Compose** | UI 开发，Material 3 设计语言 |
| **Room** | 本地数据库，photos 表（元数据） |
| **OkHttp** | HTTP 客户端，轮询 + 条件请求 |
| **NanoHTTPD** | 群主嵌入式 HTTP 服务器 |
| **Glide** | 图片加载与缓存 |
| **WifiP2pManager** | WiFi Direct P2P 连接 |
| **NsdManager** | 局域网服务发现（NSD） |
| **ZXing** | 二维码生成与解析 |
| **CameraX / ML Kit** | 扫码功能 |

### 5.3 NanoHTTPD API 端点设计

| 端点 | 方法 | 描述 |
|------|------|------|
| `/api/photos` | GET | 返回照片列表 JSON（含 ETag） |
| `/api/photos` | POST | multipart/form-data 上传照片 |
| `/api/photos/{id}` | GET | 获取照片详细信息 |
| `/api/photos/{id}` | DELETE | 删除照片（软删除） |
| `/photos/{filename}` | GET | 下载照片原图 |
| `/thumbnails/{filename}` | GET | 下载照片缩略图 |

**ETag 设计**：使用 `max(lastModified)` 的哈希作为全局版本号，响应头 `ETag: W/"v{timestamp}"`。客户端下次请求带上 `If-None-Match: W/"v{version}"`，服务端无变化返回 **304 Not Modified**。

### 5.4 Room 数据库设计（MVP）

**photos 表**：
```
id: TEXT PRIMARY KEY (UUID)
fileName: TEXT NOT NULL
remotePath: TEXT NOT NULL (群主设备上的存储路径)
thumbnailPath: TEXT (缩略图路径)
fileSize: INTEGER
mimeType: TEXT
width: INTEGER
height: INTEGER
uploaderId: TEXT NOT NULL
uploaderName: TEXT
uploadTime: INTEGER
isDeleted: INTEGER DEFAULT 0 (软删除标记)
lastModified: INTEGER
```

**users 表（简化）**：
```
id: TEXT PRIMARY KEY (设备唯一ID)
nickname: TEXT NOT NULL
lastSeen: INTEGER
```

**不提供** markers 表（第二阶段）。

### 5.5 存储规格

**存储空间检查**：采用 **「可用空间低于 max(500MB, 单次上传上限×2)」** 阈值。低于阈值时提示清理；接近耗尽时阻止新上传。**禁止写死为 5GB**。

**缓存管理**：Glide 磁盘缓存限制为 100MB，超出后自动清理最旧的缓存条目。

## 6. 性能与安全规格

### 6.1 性能指标

| 指标 | 目标 |
|------|------|
| 冷启动时间 | ≤ 2秒 |
| 热启动时间 | ≤ 0.5秒 |
| 缩略图加载（缓存） | ≤ 100ms |
| 缩略图加载（网络） | ≤ 500ms |
| 列表滚动流畅度 | 60fps |
| 内存占用（前台） | 100-200MB |

### 6.2 数据传输规格

- 传输协议：HTTP POST，multipart/form-data
- 单文件上限：50MB
- MVP 上传：整文件 + 失败重试队列
- MVP 同步：轮询 + ETag/版本条件请求（5秒间隔）
- MVP 并发：串行或有限并发（1～2路上传）

### 6.3 安全与隐私

**权限清单**：
- `INTERNET` - 网络访问
- `ACCESS_WIFI_STATE`、`CHANGE_WIFI_STATE` - WiFi Direct
- `ACCESS_FINE_LOCATION` - WiFi Direct 设备发现
- `NEARBY_WIFI_DEVICES` - Android 13+ 附近设备
- `READ_MEDIA_IMAGES` - 读取本地照片
- `CAMERA` - 扫码
- `FOREGROUND_SERVICE` - 后台上传

**隐私保护**：应用不收集任何用户数据到服务器，所有数据仅存储在参与设备本地。不使用任何分析SDK或追踪器。

## 10. 二维码载荷格式（§8.5 规范性引用）

本节为 **MVP 强制约定**，与实现计划 `docs_共享相册App_MVP_implementation_plan.md` 保持一致。

### 10.1 载荷形态

| 项目 | 约定 |
|------|------|
| **编码** | 单行文本，UTF-8（无 BOM） |
| **语义** | 载荷即群主 HTTP 服务的根地址（baseUrl） |
| **scheme** | `http`（MVP 局域网明文） |

### 10.2 URL 模板（固定）

```text
http://<host>:<port>/
```

- `<host>`：IPv4 点分十进制或可解析的主机名
- `<port>`：NanoHTTPD 监听端口（默认 8080）
- 末尾 `/`：可选，解析后规范化

**合法示例**：
```
http://192.168.49.1:8080/
http://192.168.43.1:8080
```

**非法示例（MVP 须拒绝）**：
```
192.168.49.1:8080        # 缺少 scheme
https://192.168.49.1:8080/  # scheme 错误
http://192.168.49.1/      # 缺少显式端口
```

### 10.3 解析规则

1. `trim()` 去首尾空白
2. 校验单行且非空
3. 使用 `HttpUrl.parse()` 解析；scheme 必须为 `http`，host 非空，port 为正整数
4. 规范化后与 NSD 发现路径共用同一 `baseUrl`
5. 解析失败：展示"无效的加入二维码"，不静默降级
