## 1. 背景与目标

### 1.1 问题陈述

在活动现场、旅拍、选片等场景中，多人需要在**不经过公网/云端**的前提下，快速浏览同一批照片、做挑选与标注，并仅在需要时获取原图。微信小程序在「作为被其他设备直连的服务端」方面不可靠；**原生 Android** 可在本机监听端口并结合前台服务承载会话，因此首期产品落地为 **Android 原生应用**。

### 1.2 产品愿景（首期）

在 **手机热点或同一 Wi‑Fi** 下，由创建者手机作为会话中心（Hub），其他成员通过扫码加入，实现：

- **预览优先**：默认同步与展示压缩预览图，降低等待时间与流量。
- **原图按需**：仅当用户明确执行「保存原图」等操作时，才传输并写入系统相册完整原图。
- **临时会话、无账号**：不做云端相册库、不做业务侧账号体系；会话数据随房间结束而失效，强调隐私。
- **轻量协作**：支持删除、排序、打星、备注、入选集等「选片级」操作；**不做**像素级修图。

### 1.3 成功指标（建议）

| 维度 | 指标方向（可在立项时量化） |
|------|---------------------------|
| 体验 | 百级预览图场景下，首屏可浏览时间可接受；滑动与协作操作无明显卡顿 |
| 一致性 | 多端（创建者 + 多成员）列表与元数据（星标、备注等）一致 |
| 可靠性 | 成员断线/杀进程后可凭当前二维码重进并同步最新状态 |
| 隐私 | 产品说明与权限申请与用户预期一致；无意外云端持久化 |

## 2. 范围界定

### 2.1 本期包含（In Scope）

- 局域网内星型拓扑：创建者设备运行 Session Hub，全员（含创建者）以客户端连入。
- 相册读图、预览生成与展示、按需原图拉取与写入相册。
- 房间创建、二维码加入、协作编辑与版本同步。
- 创建者结束共享或进程终止导致 Hub 停止后的「不可再入」行为。

### 2.2 本期不包含（Out of Scope）

- iOS 客户端（协议设计需预留跨端一致性，见第 8 节）。
- 云端相册、账号登录、跨会话历史漫游。
- 专业修图（曲线、图层、局部涂抹等）。
- 开机自启、后台静默常驻 Hub（与「临时会话」及隐私预期冲突，明确禁止）。

### 2.3 优化说明（相对简要的表述）

- **「严格派生命周期」与用户沟通**：对外统一说明为「创建者结束共享或创建者应用被系统结束导致服务停止后，房间不可用」；避免承诺「仅按钮结束」却在杀进程场景下无法兑现。
- **创建者教育**：在 UI/文案中提示共享期间尽量保持应用在前台或依赖前台服务通知返回，以降低 Hub 被回收概率——这是产品体验的一部分，而非仅技术备注。

## 3. 角色与权限

| 角色 | 说明 |
|------|------|
| 创建者（Host） | 发起会话、生成二维码、可结束共享并销毁房间；Hub 运行在其设备上 |
| 成员（Member） | 扫码加入；协作权限与创建者**对等**（除结束/销毁房间外） |

**权限规则（已拍板）**

- **仅创建者**可：结束共享并销毁房间。
- **创建者与成员**在协作能力上对等：删除、排序、打星、备注、入选集等（具体以消息协议与 Hub 校验为准）。

## 4. 功能需求

### 4.1 会话与房间

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-ROOM-01 | 创建者可创建房间并启动 Hub（绑定本机可访问的 IP、端口、路径） | P0 |
| FR-ROOM-02 | 房间内维护单调递增的房间版本号 `v`，状态变更后广播，客户端据此增量同步 | P0 |
| FR-ROOM-03 | 创建者显式「结束共享」后，房间关闭，已有连接断开，新连接拒绝 | P0 |
| FR-ROOM-04 | Hub 因创建者杀进程/崩溃停止时，行为与「房间结束」一致：外部不可再连 | P0 |

### 4.2 二维码与加入

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-QR-01 | 二维码 **必须在 App 内实时生成**（不依赖外部页面临时托管） | P0 |
| FR-QR-02 | 当 IP 和/或端口变化时，二维码内容更新为新的 `host`/`port`；**joinToken 从开房到结束保持不变** | P0 |
| FR-QR-03 | 提供「刷新二维码」能力（网络切换、DHCP 变更等场景） | P0 |
| FR-QR-04 | 成员可退出、断线、杀进程；**不得**因此自动销毁房间 | P0 |
| FR-QR-05 | 创建者未结束前，成员可再次扫描**当前屏幕上的有效二维码**重新进入，并同步到最新 `v` | P0 |

**二维码载荷（逻辑字段，与协议一致）**  
建议 JSON 包含：`pv`（协议版本）、`roomId`、`token`（joinToken）、`host`、`port`、`path`（WebSocket 路径，若适用）。

### 4.3 媒体：预览与原图

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-MEDIA-01 | 默认仅同步/展示预览图；预览参数可配置（如长边 720～1080 像素区间） | P0 |
| FR-MEDIA-02 | 原图仅在用户明确操作（如「保存原图」）后传输；写入系统相册需符合分区存储与权限规范 | P0 |
| FR-MEDIA-03 | 原图传输路径：经 Hub 从**持有原图的一方**分块推送至 Hub，再转发给请求方（避免各成员再开监听端口） | P0 |
| FR-MEDIA-04 | 原图通道需支持**背压**、**队列上限**、**分块大小**可配置，防止单用户拖垮 Hub | P0 |

**优化建议**

- 预览生成：本地 `Bitmap` 采样 + 压缩与 **Coil** 展示并存；大图列表注意内存与 Recycler/Lazy 复用。
- 若单图原图极大，可增加「仅 Wi‑Fi 下允许原图」或「二次确认」作为 P1 体验项（首期可不做，但在 PRD 中预留）。

### 4.4 协作（选片级）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-EDIT-01 | 支持从共享集中移除条目（删除，语义需在实现上明确：仅移出共享视图还是影响源相册，建议产品确认） | P0 |
| FR-EDIT-02 | 支持排序 | P0 |
| FR-EDIT-03 | 支持打星（或等价标记） | P0 |
| FR-EDIT-04 | 支持备注 | P0 |
| FR-EDIT-05 | 支持入选集（或标签/分组，与「入选」语义一致） | P0 |
| FR-EDIT-06 | 不做像素级修图 | 约束 |

> **需产品补充的一点**：「删除」是指共享会话内的逻辑删除，还是操作本地 MediaStore？建议在详细设计前固定语义，避免开发与验收争议。

### 4.5 网络与发现

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-NET-01 | 运行场景：同一 Wi‑Fi 或手机热点，不依赖公网传输业务数据 | P0 |
| FR-NET-02 | 创建者 UI 展示当前 Hub 监听地址信息，辅助成员排查连接问题 | P1 |

## 5. 非功能需求

### 5.1 性能与容量

- 预览同步应优先保证「先可见、后完整」；列表滚动保持流畅。
- Hub 侧应对连接数、消息速率、原图并发请求设上限，超出时降级（排队、拒绝并提示）。

### 5.2 可靠性与一致性

- 房间状态修改在 Hub 侧需 **单线程串行化** 或使用 **Mutex**，避免并发写 `RoomState`。
- 冲突策略：**LWW（Last Writer Wins）以 Hub 权威版本为准**；客户端以 `v` 驱动合并。

### 5.3 安全与隐私（首期可接受与必须告知）

| 项目 | 说明 |
|------|------|
| 传输 | 首期可接受局域网 **明文 `ws://`**，降低实现复杂度 |
| 风险披露 | 必须在隐私说明中写明：同网段内若获得 `joinToken` 可能加入房间 |
| Token | `joinToken` 须 **高熵**（足够随机、足够长），降低被猜中概率 |
| 数据留存 | 无云端业务持久化；本地缓存策略需在实现层定义（会话结束后清理） |

**可选演进（非首期）**：同一 Wi‑Fi 下的证书钉扎或 `wss`、配对 PIN，可作为后续安全增强。

### 5.4 合规与权限（Android）

- **INTERNET**、**ACCESS_NETWORK_STATE**；可选 **ACCESS_WIFI_STATE**（辅助展示 IP）。
- 读图：**READ_MEDIA_IMAGES** 及低版本兼容路径。
- 写相册：符合分区存储的 **MediaStore** 写入流程。
- **FOREGROUND_SERVICE**；Android 13+ **通知权限**。
- **禁止**使用 **BootCompleted** 自动拉起 Hub。

## 6. 技术架构（实现约束）

### 6.1 拓扑与组件

- **星型拓扑**：Session Hub 部署在创建者手机；所有客户端通过 **WebSocket** 连接 `Host IP:Port`。
- **ForegroundService**：创建者分享期间承载 Hub，展示持续通知并提供返回 App 入口。

### 6.2 推荐技术栈（与简要文档一致，可微调）

| 层次 | 选型 |
|------|------|
| UI | Kotlin + Jetpack Compose + Navigation |
| 异步 | 协程 / Flow |
| DI | Hilt（可选） |
| Hub | Ktor Server（CIO）+ WebSocket 路由 |
| 大文件 | 与 WebSocket **二选一**：同端口挂 HTTP 传文件，或统一走 WS 分块（择一并保持契约清晰） |
| 序列化 | kotlinx.serialization（JSON + `protocolVersion`） |
| 图片 | Coil；预览 Bitmap 采样 + compress |

### 6.3 协议信封（建议字段）

每条 JSON 消息建议包含：

- `pv`：协议版本  
- `type`：消息类型  
- `v`：房间版本（状态变更时单调递增并广播）  
- `cid`：客户端消息关联 ID（用于 ACK/错误对齐）  
- `payload`：业务负载  

**原图/二进制**：若走 HTTP，应对 `cid` 或资源 ID 与 WS 信令关联，避免状态机分裂。

## 7. 验收标准（可测试）

| 编号 | 场景 | 期望结果 |
|------|------|----------|
| AC-01 | 多成员加入同一房间 | 预览图可互通展示 |
| AC-02 | 成员浏览列表 | 默认不拉取原图；仅预览 |
| AC-03 | 成员执行「保存原图」 | 原图经 Hub 转发后正确写入相册（权限允许时） |
| AC-04 | 两人同时改排序/星标 | 最终以 Hub 版本为准，多端收敛一致 |
| AC-05 | 成员杀进程后重扫当前码 | 可重新进入并同步到最新状态 |
| AC-06 | 创建者点「结束共享」 | 他人不可再连；房间失效 |
| AC-07 | 创建者杀进程导致 Hub 停 | 他人不可再连（与对外说明一致） |
| AC-08 | IP/端口变化后刷新二维码 | 新码可连；**token 不变** |

## 8. 跨平台边界（iOS 后续）

- 首期 **不做 iOS**。
- Android 侧协议（`pv`、消息类型、二维码 JSON 语义）视为 **跨端契约**；后续 iOS 客户端应复用同一套语义以降低双端成本。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 创建者应用被系统杀进程 | 前台服务 + 用户教育；对外说明与「严格派生命周期」一致 |
| 局域网 IP 频繁变化 | 刷新二维码；UI 提示网络变化 |
| 原图打满带宽/内存 | 背压、队列上限、分块大小配置、并发限制 |
| 明文 WS 被同网窃听 | 隐私说明 + 高熵 token；后续可迭代加密或 wss |

---

## 10. 附录：一句话收束

**产品**：热点/同 Wi‑Fi 下的临时共享相册，预览为主、原图按需，协作涵盖删/排/星/注/入选；创建者结束或关应用即失效；成员可反复扫当前码重进。

**技术**：Kotlin + Compose + Ktor（CIO）在创建者手机做 Hub + 前台服务，成员 WebSocket 接入；原图经 Hub 分块转发；状态以版本号 `v` 驱动，冲突 LWW。
