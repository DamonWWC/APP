一、已确定的产品需求（Android 首期）
1. 场景与目标
局域网：手机热点或同一 Wi-Fi，不依赖公网传照片（微信打开小程序仍可能依赖网络，本期讨论落地为 原生 Android App）。
快速共享大量照片：以预览图为主，减少等待与流量。
按需原图：只有用户点「保存原图」等明确操作后，才传/存完整原图到系统相册。
无云端业务持久化：不做云端相册库、不做账号体系；会话数据临时，强调隐私。
协作（首期范围）：挑选类编辑即可——删除、排序、打星、备注、入选集；不做像素级修图。
权限：除「仅创建者可结束共享并销毁房间」外，创建者与成员协作权限对等。
2. 会话与二维码（已拍板规则）
二维码：必须在 App 内实时生成；IP/端口变化时只更新地址；joinToken 从开房到结束不变（「永久到结束」）；提供「刷新二维码」。
成员：可退出、断线、杀进程；不会因此销毁房间。
重进：创建者未结束前，成员可再次扫当前屏幕上的码重新进入并同步最新状态。
严格派生命周期（对外说法）：创建者点「结束共享」或关闭/杀死应用导致 Hub 停 → 其他人无法再进入（与「只有点按钮才结束」在「杀进程」上对齐现实：进程没了服务也没了）。
创建者使用预期：共享期间尽量前台或符合规范的前台服务，降低 Hub 被系统回收概率。
3. 验收方向（摘要）
预览互通；原图仅按需；协作多端一致；成员可杀进程后重进；创建者结束或杀进程后不可再连；IP 变后刷新码可进且 token 不变。
二、已确定的 Android 技术路线（方法）
1. 总体架构（最稳妥、讨论结论）
星型拓扑：创建者手机上跑 Session Hub（内嵌服务）；所有人（含创建者自己） 以 WebSocket 客户端连到 Host 的 IP:端口。
原因：微信小程序不能可靠地当「被别的手机直连的服务端」；原生 Android 可以在本机监听端口 + 前台保活，因此闭环最稳。
原图首期推荐：经 Hub 从「持有原图的一方」分块推到 Hub，再转发给请求方（避免每台成员再开监听端口）；配合背压、队列上限、分块大小配置。
2. Android 技术栈（建议）
Kotlin + Jetpack Compose + Navigation
协程 / Flow；Hilt（可选）
Hub：Ktor Server（CIO） + webSocket 路由（可选同端口挂 HTTP 传文件，二选一统一）
序列化：kotlinx.serialization（JSON 消息 + protocolVersion）
图片：Coil 显示预览；Bitmap 采样 + compress 生成预览图（长边约 720～1080 一类参数可配置）
相册：ContentResolver 读图；MediaStore 写原图；处理好读/写权限与分区存储
3. Android 组件与系统行为
ForegroundService：创建者分享期间承载 Hub（持续通知、返回 App 入口）。
不要用 BootCompleted 自动拉起 Hub（与「临时会话」和隐私预期不符）。
权限：INTERNET、ACCESS_NETWORK_STATE、读图（READ_MEDIA_IMAGES 及低版本兼容）、写相册、FOREGROUND_SERVICE、Android 13+ 通知权限；可选 ACCESS_WIFI_STATE 辅助展示 IP。
线程：Hub IO 走协程；RoomState（房间状态） 修改需单线程或 Mutex，避免并发写状态。
4. 协议与数据（实现契约）
消息：JSON 信封建议含 pv、type、v（房间版本）、cid、payload；状态变更使 v 单调递增并广播。
二维码 JSON：pv、roomId、token、host、port、path。
冲突：LWW（以 Hub 版本为准）。
首期安全：可接受局域网 明文 ws://，但要在隐私说明里写清「同网段拿到 token 可加入」的风险；joinToken 要高熵。
5. 与 iOS 的关系（仅作边界）
首期不做 iOS；后续若做，保持 pv、消息类型、二维码 JSON 语义 一致即可，Android 侧协议即「跨端契约」。
三、一句话收束（Android）
产品：热点/同 Wi-Fi 下的临时共享相册，预览为主、原图按需，协作做删/排/星/注/入选，创建者结束或关应用即失效，成员可反复扫当前码重进。
技术：Kotlin + Compose + Ktor(CIO) 在创建者手机做 Hub + 前台服务，成员 WebSocket 接入，原图首期建议 经 Hub 分块转发，状态 version + LWW。
