# 局域网临时共享相册 — Android 首期详细需求文档（Wi‑Fi Direct 连接版）

> **文档版本**：2.0（Wi‑Fi Direct）  
> **基线**：由《详细需求文档》v1.0 改写，**连接与发现层**由「同 Wi‑Fi / 手机热点 + 局域网 IP」改为 **Wi‑Fi Direct（Wi‑Fi P2P）**；业务会话、协作与协议语义尽量与 v1.0 对齐。  
> **范围**：Android 原生应用首期；不含 iOS、不含云端业务  

---

## 1. 文档说明

### 1.1 目的

在 v1.0 已拍板的产品结论基础上，将**设备互联方式**明确为 **Wi‑Fi Direct**：在**无路由器、无手机热点 AP**（或热点不可用的场景）下，通过 **P2P 组网**建立二层链路，再在其上承载原有 **Session Hub + WebSocket** 架构。本文档保持**可交付、可验收、可开发**的粒度，并单独说明 P2P 带来的权限、发现、Group Owner（GO）与地址约定等实现要点。

### 1.2 读者

- 产品与交互设计  
- Android 开发 / 测试  
- 后续若扩展 iOS，需注意 **iOS 无官方对等 Wi‑Fi Direct API**，跨端需另定通道（见第 10 章）  

### 1.3 名词表

| 术语 | 含义 |
|------|------|
| **创建者 / Host** | 发起共享会话、本机运行 Session Hub 的用户；在 P2P 中通常为 **Group Owner（GO）** 或经产品定义为固定「服务端角色」的一方 |
| **成员 / Member** | 加入会话的其他用户；在 P2P 中通常为 **Client** |
| **Wi‑Fi Direct / P2P** | IEEE 802.11 设备间直连与组网能力；Android 通过 `WifiP2pManager` 等 API 暴露 |
| **GO（Group Owner）** | P2P 组内的「软 AP」角色，为组内分配 IP（典型实现下 GO 侧有固定或可预期的网内地址） |
| **P2P Client** | 加入 GO 所建组的设备，从 GO 获得组内 IPv4 地址 |
| **Session Hub** | 运行在创建者设备上的内嵌服务，负责 WebSocket 接入与房间状态权威（与 v1.0 相同） |
| **房间 / Room** | 一次共享会话的逻辑边界，有唯一 `roomId` 与生命周期 |
| **joinToken** | 加入房间所需的高熵凭证，从开房到结束保持不变 |
| **房间版本 v** | 单调递增的状态版本号，用于同步与冲突处理 |
| **预览图** | 降采样/压缩后的缩略图 |
| **原图** | 用户设备上的完整分辨率图片数据（按需传输） |
| **deviceAddress** | P2P 设备 MAC 地址字符串（如 `aa:bb:cc:dd:ee:ff`），用于定向连接 |

## 2. 产品愿景与首期目标

### 2.1 核心价值

在 **Wi‑Fi Direct 已组网成功** 的前提下，多人在**不依赖公网相册/账号、不依赖现有 Wi‑Fi 路由器或手机热点 SSID** 的情况下，快速共享大量照片：**以预览为主、原图按需**，并完成同一会话内的**挑选类协作**（删、排、星、备注、入选集等），强调**临时性与隐私**。

**与 v1.0 的差异（连接层）**：

- **v1.0**：同网段可达（同 Wi‑Fi 或创建者开热点），通过 **IP:端口** 连 Hub。  
- **v2.0（本文档）**：成员设备需先与创建者设备完成 **P2P 连接/入组**，再在组内 IP 上访问 Hub（仍为 TCP/WebSocket；Hub 实现可与 v1.0 同为 Ktor 等）。

### 2.2 为何首期仍为原生 Android App

与 v1.0 相同：创建者设备需长期扮演 **Hub 服务端** 并结合**前台服务**保证存活；Wi‑Fi Direct 的 **GO + 监听端口** 在原生侧可控性最好。

### 2.3 首期不做（Out of Scope）

- 云端相册库、账号体系、跨设备历史会话恢复  
- 像素级修图  
- **iOS 客户端通过 Wi‑Fi Direct 与 Android 对等组网**（系统能力限制，不作首期承诺）  
- 创建者杀进程后房间仍「幽灵在线」  
- **跨品牌/跨系统 P2P 互通保证**（首期以 Android 同应用为准；不同 OEM 行为差异见风险章）

## 3. 用户场景与故事

### 3.1 典型场景

1. **无外网/无路由器现场**：创建者开共享，系统建立 P2P 组（GO 在创建者侧），成员通过扫码或应用内发现加入 P2P 组，再自动连上 Hub，浏览预览与协作。  
2. **不便开手机热点**：热点与运营商策略冲突或用户不愿暴露热点 SSID 时，仍可用直连。  
3. **省流量与按需原图**：与 v1.0 相同，默认仅预览，原图经明确操作再传。

### 3.2 用户故事（摘要）

- 作为**创建者**，我希望在开启共享后，他人能通过**扫码或应用内引导**完成 **Wi‑Fi Direct 连接**，然后自动进入房间；**P2P 断链或 GO 重建**时，界面能提示重新连接或刷新凭证。  
- 作为**成员**，我希望在 **P2P 已连接** 的前提下，断线重连 WebSocket、杀进程后，在创建者未结束会话时，能按产品引导**重新完成 P2P + 加入**。  
- 作为**任一方**，我希望只有明确操作才下载完整原图。  
- 作为**创建者**，只有我结束会话或应用退出导致 Hub 停止时，会话终结。

## 4. 功能需求

### 4.1 会话生命周期（含 P2P 阶段）

| 阶段 | 行为说明 |
|------|----------|
| **P2P 准备** | 创建者选择照片范围，发起「开始共享」；系统创建或复用 P2P 组，创建者侧为 GO（推荐固定策略，见第 6 章） |
| **发现与入组** | 成员通过 **二维码 / 近场发现（可选）** 获取创建者 P2P 信息，发起 `connect`；连接成功后双方处于同一 P2P 二层网络 |
| **应用会话创建** | P2P 就绪后，创建者启动（或保持）Session Hub 监听；展示含 `joinToken` 与连接参数的二维码或同等信息 |
| **进行中** | 全员（含创建者本机客户端）以 **WebSocket 客户端** 连 Hub；状态同步规则同 v1.0 |
| **结束（正常）** | 创建者「结束共享」→ 停止 Hub → 断开/移除 P2P 组（策略可配置，建议结束即拆组释放无线资源） |
| **结束（异常）** | 创建者关闭或杀死应用 → Hub 与 P2P 服务停止 → 他人无法再连，与 v1.0 一致 |

**产品表述建议**：对外说明包含两层：**「直连 Wi‑Fi（P2P）」+「应用共享服务」**；任一关键层终止，会话即不可用。

### 4.2 二维码 / 入会载荷规则（Wi‑Fi Direct 适配）

- **生成位置**：App 内**实时生成**；当 **P2P 组信息、Hub 监听地址或端口** 变化时，应刷新展示。  
- **joinToken**：从开房到会话结束**不变**（与 v1.0 一致）。  
- **手动刷新**：提供「刷新」入口：重新生成 P2P 组信息展示、重新探测本机 GO 侧 **Hub 绑定地址/端口** 并更新二维码。  
- **建议二维码 JSON 逻辑字段**（在 v1.0 的 `pv`、`roomId`、`token`、`port`、`path` 基础上扩展）：

| 字段 | 含义 |
|------|------|
| `pv` | 协议 / 应用协议版本 |
| `roomId` | 房间 id |
| `token` | joinToken |
| `port` | Hub 监听端口 |
| `path` | WebSocket 路径（若有） |
| **`goDeviceAddress`** | 创建者 P2P 设备 MAC，供成员 `WifiP2pConfig.deviceAddress` 定向连接 |
| **`ssid` / `psk`**（可选） | 若实现选择「展示组密码型」入会（部分流程需用户手动连热点式 UI），则加密存储与展示策略需单独立项；**推荐首期以应用内 P2P API 自动连接为主，减少明文 PSK 展示** |
| **`hubHost`**（可选） | 若实现采用固定约定（如 GO 侧 `192.168.49.1`），可写入以便客户端校验；若采用运行时探测，则以 App 内逻辑为准，二维码可省略 |

**优化建议**：UI 明确提示：「请开启 Wi‑Fi 与定位相关权限」「请将两台设备靠近」「若无法连接请刷新二维码并让成员重新扫码」。

### 4.3 成员侧行为

- 成员主动退出应用、断 P2P、断 WebSocket：**不销毁**房间。  
- **重进条件**：创建者未结束且 Hub 仍在；成员需 **P2P 再次连通** 后，使用**当前屏幕上的**二维码或保存的会话凭证（若产品允许）重新加入。  
- 重进后**全量或增量同步**至最新 `v`（与 v1.0 相同）。

### 4.4 权限模型

与 v1.0 相同：仅创建者可结束共享；其余协作对等。

### 4.5 协作能力（首期）

与 v1.0 相同：删除（建议默认会话内移除）、排序、打星、备注、入选集。

### 4.6 图片与流量策略

与 v1.0 相同：预览为主，原图经 Hub 分块转发，背压与队列上限必选。

### 4.7 隐私与合规展示

- 无云端持久化：临时会话数据，结束后清理策略与 v1.0 一致。  
- **首期安全**：P2P 链路与 Hub 仍可采用 **`ws://` 明文**（与 v1.0 同级），须在隐私说明中写明：  
  - **同 P2P 组内**若 `joinToken` 泄露，第三方设备在已入组前提下可能加入会话；  
  - Wi‑Fi Direct 为无线介质，需告知用户仅在可信场景使用。  
- **joinToken**：高熵；**`goDeviceAddress` 非密钥**，须配合 token 校验。  

**优化建议（P1）**：应用层 TLS 或二进制封装升级路径，避免协议字段写死无法演进。

## 5. 非功能需求

### 5.1 性能与体验

- 大量照片下列表与预览流畅；**P2P 连接耗时**需有明确 Loading 与失败重试文案。  
- **GO 设备**承担组内转发，原图并发与预览广播策略需更保守（见风险章）。

### 5.2 稳定性

- Hub 的 FD/内存限制与 v1.0 相同。  
- **P2P 回调与 Hub 网络线程**需统一生命周期：GO 变更、组移除时主动断开所有 WebSocket 并提示重连。

### 5.3 兼容性（Android + Wi‑Fi Direct）

- **读图 / 写相册**：与 v1.0 相同（`READ_MEDIA_IMAGES`、分区存储等）。  
- **前台服务**：创建者分享期间须持有前台服务承载 Hub + P2P 相关长连接逻辑（具体类型按 targetSdk 选择合规的 `foregroundServiceType`）。  
- **Android 13（API 33）及以上**：`NEARBY_WIFI_DEVICES` 与精确位置权限策略需按官方文档实现（扫描 P2P 设备通常仍涉及位置权限或新附近设备权限组合）。  
- **禁止**：`BootCompleted` 自动拉起 Hub（与 v1.0 相同）。

### 5.4 可维护性

- 协议含 `pv`；二维码 JSON 含 `pv`，便于扩展 **P2P 字段** 而不破坏旧客户端解析（未知字段忽略）。

## 6. Android 技术方案（Wi‑Fi Direct + Hub）

### 6.1 拓扑与数据流

- **星型拓扑（逻辑不变）**：Session Hub 运行于创建者手机；所有人（含创建者本机）以 **WebSocket 客户端**连接 Hub 的 `hubHost:port`。  
- **物理层**：全员先处于同一 **Wi‑Fi Direct Group**；IP 分配由 GO 侧系统实现（Android 常见为 GO `192.168.49.1`，客户端 `192.168.49.0/24` 内地址——**以实机探测为准，文档允许实现侧固化约定并预留探测 fallback**）。  
- **原图路径**：与 v1.0 相同：持有方 → 分块至 Hub → 转发至请求方。  
- **Hub 实现**：仍推荐 **Ktor Server（CIO）+ WebSocket**；监听地址应绑定在 **P2P 网络接口**（避免误绑蜂窝/VPN 接口导致成员不可达）。

### 6.2 技术栈建议

| 层级 | 选型 |
|------|------|
| UI | Kotlin + Jetpack Compose + Navigation |
| 异步 | 协程 / Flow |
| DI | Hilt（可选） |
| **P2P** | **`WifiP2pManager` + `BroadcastReceiver`（或官方推荐的新 API 若迁移）**；连接意图与状态机单独封装 |
| Hub | Ktor Server（CIO），WebSocket 路由 |
| 序列化 | kotlinx.serialization |
| 图片 | Coil；预览采样与压缩 |
| 相册 | ContentResolver / MediaStore |

### 6.3 P2P 关键实现要点（需求级，非代码）

1. **GO 角色**：首期建议 **创建者固定为 GO**（在 `createGroup` / `connect` 参数中满足「创建者创建组」的流程），减少「成员误为 GO 导致 Hub IP 变化」的支持成本。  
2. **Hub 启动时机**：建议在 **P2P 组已建立且本机接口 IP 可用** 后再 `start()`；若先起 Hub 后 P2P，须在接口可用时**重启监听或迁移绑定**（实现二选一，需在技术设计书中闭环）。  
3. **成员连接顺序**：**先 P2P `connect` 成功**，再使用解析自二维码的 `hubHost`（或约定 IP）+ `port` 建立 WebSocket。  
4. **断链**：P2P 断链 → 主动关闭 WebSocket → UI 提示重新入组；Hub 可不立即销毁房间（与 v1.0「成员断线不销毁房间」一致），但成员侧必须重新 P2P 才能再连。

### 6.4 权限清单（Wi‑Fi Direct 相关摘要）

除 v1.0 的 `INTERNET`、`ACCESS_NETWORK_STATE`、存储/相册、前台服务、通知外，至少评估并声明：

- `ACCESS_WIFI_STATE`、`CHANGE_WIFI_STATE`  
- **`ACCESS_FINE_LOCATION`**（或按 targetSdk 使用 **`NEARBY_WIFI_DEVICES`** 等新模型；以扫描 P2P  peers 的合规要求为准）  
- 厂商差异：部分机型对 P2P 并发、省电限制更严，需在测试矩阵中覆盖。

## 7. 协议与数据契约

### 7.1 消息信封

与 v1.0 相同：`pv`、`type`、`v`、`cid`、`payload`。

### 7.2 状态同步与冲突

与 v1.0 相同：Hub 权威，LWW，`v` 单调递增广播。

### 7.3 二维码 JSON（逻辑）

**最小集**：`pv`、`roomId`、`token`、`port`、`path`、`goDeviceAddress`。  
可选：`hubHost`（若固定约定或探测结果写入）。

### 7.4 跨端预留（iOS）

首期不做 iOS。若未来需要 iOS 参与同一产品，需知：**iOS 不具备与 Android 对等的 Wi‑Fi Direct 编程模型**，通常需改为 **Multipeer Connectivity / 同局域网** 等替代链路；**应用层消息 `pv`、`type`、`v` 语义**仍可复用，**物理发现层**将分叉。

## 8. 验收标准（可测试）

| 编号 | 验收项 |
|------|--------|
| AC-1 | 多端在 **P2P 已连接** 前提下，可同时浏览**预览**，内容与协作状态一致 |
| AC-2 | 未触发「保存原图」等明确操作时，不拉取完整原图 |
| AC-3 | 协作操作与 Hub `v` 广播一致 |
| AC-4 | 成员杀进程或断 WebSocket 后，在创建者未结束且 **P2P 仍可用或已重连** 前提下，可按**当前**二维码重新加入并同步到最新 `v` |
| AC-5 | 创建者结束共享后，他人无法再连 Hub；P2P 组按设计释放 |
| AC-6 | 创建者杀进程导致服务停止后，他人无法再连 |
| AC-7 | **P2P 组重建或 Hub 监听地址变化**后，创建者刷新二维码，成员扫新码可完成 **P2P + 入会**；**token 不变** |
| AC-8 | 隐私说明含：明文传输风险、token 泄露风险、无线直连使用场景提示 |
| **AC-9** | **首次使用或系统要求时**，权限申请与说明文案完整，且在不授权时行为可解释（不可用而非静默失败） |

## 9. 里程碑与优先级建议

### 9.1 P0（首期必达）

- **P2P 建组 / 定向连接 / 断线提示**  
- **Hub 仅在 P2P 接口可达** 场景下可被成员访问  
- 扫码加入（含 `goDeviceAddress`）+ WebSocket 稳定连接  
- 预览同步、前台服务、结束逻辑  
- 「保存原图」按需传输  
- 协作：删、排、星、备注、入选集（最小集合可按迭代裁剪，但合同建议写清）  

### 9.2 P1（增强）

- 应用内「附近设备」列表发现（减少扫码步骤）  
- P2P + WebSocket 双重断线重连策略细化  
- 原图取消、进度、重试  
- 安全升级（TLS 等）

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| **GO 角色落到成员机** 导致 Hub 不可达 | 产品固定「创建者创建组并优先 GO」；技术侧校验本机是否为 GO 再启动 Hub |
| **不同 OEM P2P 行为差异** | 建立机型测试矩阵；失败码映射为用户可懂文案与重试路径 |
| **Android 位置/附近设备权限引发投诉** | 首次说明「用于 Wi‑Fi 直连发现设备」；遵循最小权限 |
| **GO 侧 CPU/射频负载高** | 限制原图并发、预览尺寸、广播频率 |
| **用户误以为「不用连 Wi‑Fi」** | 文案写清：需开启设备 Wi‑Fi 硬件（非连路由器） |
| token 泄露 | 高熵 token；短期会话；长期 TLS |

## 11. 附录：一句话收束

**产品**：通过 **Wi‑Fi Direct** 在设备间建立临时无线组网，在组内访问创建者手机上的 **Session Hub**，实现预览为主、原图按需的共享相册与协作；创建者结束或退出导致服务停止即会话失效。  

**技术**：Kotlin + Compose；**`WifiP2pManager` 管理 P2P 组与连接**；创建者侧 **Ktor（CIO）Hub + 前台服务**；全员 **WebSocket**；原图经 Hub 分块转发；**`v` + Hub 权威 LWW** 同步。

| 二层连通 | 同网段路由或热点 DHCP | **Wi‑Fi Direct Group** |
| 发现与地址 | IP、端口为主 | **P2P 连接 + 组内 IP（常配合 GO 地址约定）** |
| 二维码 | `host`+`port`+`token`… | **`goDeviceAddress` 必备**；`hubHost` 可选 |
| 权限 | 网络、存储、前台服务 | 另加 **P2P/定位或附近设备相关权限** |
| iOS 预留 | 协议一致即可扩 | **直连层在 iOS 需替换实现** |
