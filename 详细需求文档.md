# 局域网临时共享相册 — Android 首期详细需求文档

> **文档版本**：1.0  
> **状态**：基于《简要需求文档》扩展与优化  
> **范围**：Android 原生应用首期；不含 iOS、不含云端业务

---

## 1. 文档说明

### 1.1 目的

本文档在已拍板的产品与技术结论基础上，补充**可交付、可验收、可开发**的细化说明，统一术语，并标注与原文相比的**优化建议**（非强制变更项将单独标明）。

### 1.2 读者

- 产品与交互设计
- Android 开发 / 测试
- 后续若扩展 iOS，可作为跨端协议基准

### 1.3 名词表

| 术语 | 含义 |
|------|------|
| **创建者 / Host** | 发起共享会话、本机运行 Session Hub 的用户 |
| **成员 / Member** | 通过扫码加入会话的其他用户 |
| **Session Hub** | 运行在创建者设备上的内嵌服务，负责 WebSocket 接入与房间状态权威 |
| **房间 / Room** | 一次共享会话的逻辑边界，有唯一 `roomId` 与生命周期 |
| **joinToken** | 加入房间所需的高熵凭证，从开房到结束保持不变 |
| **房间版本 v** | 单调递增的状态版本号，用于同步与冲突处理 |
| **预览图** | 降采样/压缩后的缩略图，用于列表与快速浏览 |
| **原图** | 用户设备上的完整分辨率图片数据（按需传输） |

## 2. 产品愿景与首期目标

### 2.1 核心价值

在**手机热点或同一 Wi‑Fi** 下，多人在**不依赖公网相册/账号**的前提下，快速共享大量照片：**以预览为主、原图按需**，并在同一会话内完成**挑选类协作**（删、排、星、备注、入选集等），强调**临时性与隐私**。

### 2.2 为何首期落地为原生 Android App

微信小程序难以稳定承担「被其他手机直连的服务端」角色；原生 Android 可在本机**监听端口**并结合**前台服务**承载 Hub，闭环最稳。首期明确**不做**依赖微信打开小程序作为主路径。

### 2.3 首期不做（Out of Scope）

- 云端相册库、账号体系、跨设备历史会话恢复
- 像素级修图（滤镜、曲线、图层等）
- iOS 客户端（协议需预留跨端一致性，见第 10 章）
- 创建者杀进程后房间仍「幽灵在线」（与严格派生命周期一致，见 4.3）

## 3. 用户场景与故事

### 3.1 典型场景

1. **活动/外拍现场**：创建者开热点，众人加入，快速浏览预览并标星、备注、整理入选集。  
2. **小团队选片**：多人同时调整顺序与删除废片，状态实时一致。  
3. **省流量**：默认只同步预览；需要打印/精修时再「保存原图」到本地相册。

### 3.2 用户故事（摘要）

- 作为**创建者**，我希望开房后生成二维码，便于他人加入；网络变化时能刷新二维码继续加入。  
- 作为**成员**，我希望断线、杀进程后仍能扫**当前屏幕上的码**重新进入并看到最新协作结果。  
- 作为**任一方**，我希望只有明确操作才下载完整原图，避免流量与等待。  
- 作为**创建者**，我希望只有我（或我结束会话/应用退出导致 Hub 停止）能终结会话；成员退出不销毁房间。

## 4. 功能需求

### 4.1 会话生命周期

| 阶段 | 行为说明 |
|------|----------|
| 创建 | 创建者选择要共享的照片（或相册范围），启动 Hub，生成 `roomId`、`joinToken`，展示二维码 |
| 进行中 | 成员通过扫码 + token 校验加入；全端通过 WebSocket 同步状态 |
| 结束（正常） | 创建者点击「结束共享」→ Hub 停止 → 房间失效，他人不可再连 |
| 结束（异常/现实对齐） | 创建者**关闭或杀死应用**导致 Hub 进程结束 → 与上条等效，**他人不可再连** |

**产品表述建议**：对外说明统一为「创建者结束共享或创建者应用退出导致服务停止，会话即结束」，避免「仅按钮结束」与「杀进程」说法冲突。

**创建者侧预期**：共享期间尽量保持应用在前台，或依赖符合规范的前台服务，以降低系统回收概率（不保证 100% 不被杀，此为系统行为）。

### 4.2 二维码规则（已拍板 + 实现要点）

- **生成位置**：必须在 App 内**实时生成**（非静态图缓存过期仍展示）。  
- **内容变化**：当 **IP 和/或端口** 变化时，二维码中**仅更新 `host`/`port`（及必要时 `path`）**；**`joinToken` 从开房到会话结束保持不变**（「永久到结束」）。  
- **手动刷新**：提供「刷新二维码」入口，重新读取当前网络地址并刷新展示。  
- **二维码载荷（逻辑）**：建议 JSON 包含：`pv`、`roomId`、`token`、`host`、`port`、`path`（与协议章一致）。

**优化建议（可选）**：在 UI 上提示「若他人无法连接，请尝试刷新二维码或确认同一局域网/热点」，降低支持成本。

### 4.3 成员侧行为

- 成员可**主动退出**、**断线**、**杀进程**：均**不销毁**房间。  
- **重进条件**：创建者未结束会话且 Hub 仍在；成员可再次扫描**创建者当前屏幕上的二维码**重新加入。  
- 重进后应**全量或增量同步**至最新 `v` 对应的房间状态（实现策略见技术章）。

### 4.4 权限模型

- **唯一不对称点**：仅创建者可**结束共享并销毁房间**（含 Hub 停机带来的不可再加入）。  
- **其余协作**：创建者与成员**权限对等**（删除、排序、打星、备注、入选集等，见 4.5）。

### 4.5 协作能力（首期）

| 能力 | 说明 |
|------|------|
| 删除 | 从会话视图中移除条目（需明确：是否仅「会话内删除」不影响对方系统相册源文件——**建议默认：仅会话集合内移除，不删对方设备文件**） |
| 排序 | 调整照片在共享列表中的顺序 |
| 打星 | 星级或标记状态 |
| 备注 | 文本备注，随状态同步 |
| 入选集 | 将子集标记为入选，便于筛选 |

**不做**：像素级修图、云端持久协作历史。

**优化建议（建议采纳为默认规则）**：在需求中明确「删除」语义为**会话集合内移除**，避免用户误以为会删除他人手机相册文件，引发纠纷。

### 4.6 图片与流量策略

- **默认**：同步与展示以**预览图**为主（生成参数可配置，如长边约 720～1080）。  
- **原图**：仅在用户执行**「保存原图」**或产品定义的同等明确操作后，才从**持有原图的一方**经 Hub **分块传输**至请求方，并写入系统相册（权限与分区存储需合规）。  
- **背压**：原图通道应有队列上限、分块大小与超时策略，避免 Hub 内存暴涨或拖垮创建者手机。

### 4.7 隐私与合规展示

- 无云端业务持久化：会话数据**临时**；结束后以产品设计为准清理本地缓存（建议写明保留策略，如「结束即清」或「仅内存」）。  
- **首期安全**：可接受局域网 **明文 `ws://`**，但必须在**隐私说明 / 首次说明**中写明：**同网段内若 token 泄露，第三方可能加入会话**。  
- **joinToken**：须**高熵**（足够随机、足够长度）。

**优化建议（P1）**：若同网段存在更高安全诉求，可预留 **WSS/TLS** 或应用层加密切换开关（首期可不做，但协议设计时避免把「明文」写死到无法升级）。

## 5. 非功能需求

### 5.1 性能与体验

- 大量照片场景下，列表滚动与预览加载应流畅；预览生成与传输应**异步**，避免阻塞 UI。  
- 网络抖动时：断线重连、提示可理解；重进后状态一致。

### 5.2 稳定性

- Hub 在创建者设备上长期运行时，需考虑内存与 FD 限制；原图转发必须有上限与取消策略。  
- `RoomState` 修改需**单线程或 Mutex**，避免并发写导致状态错乱。

### 5.3 兼容性

- 读图：`READ_MEDIA_IMAGES` 及低版本兼容路径；写相册与分区存储（Scoped Storage）合规。  
- Android 13+ 通知权限；前台服务类型与系统要求随 targetSdk 适配。

### 5.4 可维护性

- 协议含 **`protocolVersion`（pv）**，便于后续扩展字段与兼容旧客户端。

## 6. Android 技术方案（已确定路线 + 细化）

### 6.1 拓扑与数据流

- **星型拓扑**：Session Hub 运行于创建者手机；**所有人（含创建者自己）**均以 **WebSocket 客户端**连接 `host:port`。  
- **原图首期路径**：持有方 → **分块推至 Hub** → Hub **转发**给请求方（避免每台成员再开监听端口）。  
- **与 HTTP 二选一**：Hub 使用 **Ktor Server（CIO）** + WebSocket；可选同端口挂 HTTP 传文件，与团队选定的一种大文件策略**统一**即可。

### 6.2 技术栈建议

| 层级 | 选型 |
|------|------|
| UI | Kotlin + Jetpack Compose + Navigation |
| 异步 | 协程 / Flow |
| DI | Hilt（可选） |
| Hub | Ktor Server（CIO），WebSocket 路由 |
| 序列化 | kotlinx.serialization（JSON + `protocolVersion`） |
| 图片展示 | Coil |
| 预览生成 | Bitmap 采样 + compress，参数可配置 |
| 相册 | ContentResolver 读；MediaStore 写原图 |

### 6.3 系统组件与策略

- **ForegroundService**：创建者分享期间承载 Hub；持续通知，提供返回 App 入口。  
- **禁止**：使用 `BootCompleted` **自动拉起** Hub（与临时会话、隐私预期不符）。

### 6.4 权限清单（摘要）

- `INTERNET`、`ACCESS_NETWORK_STATE`  
- 读图相关（含旧版存储权限兼容）  
- 写相册  
- `FOREGROUND_SERVICE`（及对应前台服务类型）  
- Android 13+ 通知权限  
- 可选：`ACCESS_WIFI_STATE`（辅助展示 IP）

## 7. 协议与数据契约（实现向）

### 7.1 消息信封（建议字段）

JSON 消息建议包含：

| 字段 | 含义 |
|------|------|
| `pv` | 协议 / 应用协议版本 |
| `type` | 消息类型 |
| `v` | 房间状态版本（状态变更时单调递增并广播） |
| `cid` | 客户端消息 id（用于 ACK、去重、排错） |
| `payload` | 业务负载 |

### 7.2 状态同步与冲突

- 任一**状态变更**导致 `v` 单调递增，并由 Hub **广播**给各连接。  
- **冲突解决**：**LWW（Last Write Wins）以 Hub 权威为准**（客户端本地仅作乐观 UI 时需可被 Hub 版本覆盖）。

### 7.3 二维码 JSON（逻辑）

与简要文档一致：`pv`、`roomId`、`token`、`host`、`port`、`path`。

### 7.4 跨端预留（iOS）

首期不做 iOS；后续若做，应保持 **`pv`、消息 `type` 集合、二维码 JSON 语义** 一致，Android 侧协议即**跨端契约**。

## 8. 验收标准（可测试）

| 编号 | 验收项 |
|------|--------|
| AC-1 | 多端可同时浏览**预览**，内容与顺序、标记一致 |
| AC-2 | 未触发「保存原图」等明确操作时，不应拉取完整原图（可通过流量/日志/代理辅助验证） |
| AC-3 | 协作操作（删、排、星、备注、入选）在**多端一致**，且与 Hub `v` 递增广播一致 |
| AC-4 | 成员杀进程或断网后，在创建者未结束前提下，扫**当前**二维码可重进并同步到最新状态 |
| AC-5 | 创建者点「结束共享」后，他人无法再连接；状态符合产品设计 |
| AC-6 | 创建者杀进程或强退导致 Hub 停止后，他人无法再连接（与 AC-5 一致） |
| AC-7 | IP/端口变化后，创建者刷新二维码，成员扫新码可加入；**token 不变** |
| AC-8 | 隐私说明中包含：局域网明文风险、同网段 token 泄露风险 |

## 9. 里程碑与优先级建议

### 9.1 P0（首期必达）

- 开房 / 扫码加入 / WebSocket 稳定连接  
- 预览同步 + 基础列表  
- 前台服务 + Hub 生命周期与结束逻辑  
- 「保存原图」按需传输 + 相册写入  
- 协作：删除、排序、打星、备注、入选集（最小可用集合可再拆迭代，但合同上建议写清首期包含项）

### 9.2 P1（增强）

- 原图传输取消、进度与失败重试体验  
- 更完善的断线重连与状态补偿  
- 可选安全升级路径（TLS / 令牌轮换策略——若与「token 永久到结束」冲突则仅作下期）

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| 创建者 Hub 被系统杀 | 前台服务 + 用户教育；产品说明「结束/退出即停」 |
| 局域网 IP 频繁变化 | 刷新二维码；UI 提示检查网络 |
| 原图过大拖垮 Hub | 分块、队列上限、背压、并发限制 |
| token 被同网段截获 | 说明风险；高熵 token；长期考虑 TLS |

## 11. 附录：一句话收束

**产品**：热点/同 Wi‑Fi 下的临时共享相册，预览为主、原图按需，协作涵盖删/排/星/注/入选；创建者结束或应用退出导致 Hub 停止即会话失效；成员可反复扫当前码重进。  

**技术**：Kotlin + Compose + Ktor（CIO）在创建者手机部署 Hub + 前台服务，全员 WebSocket 接入；原图经 Hub 分块转发；状态以版本号 `v` + Hub 权威 LWW 同步。
